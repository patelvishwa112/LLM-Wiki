## The Big-Picture Mental Model

MCP is a layered system:

> **Host orchestrates → Client communicates → Server exposes capabilities → Backend systems do the real work**

MCP separates AI interaction from capability access, and capability access from backend execution.

## Core Components

### 1. User
Origin of intent. Does not interact with MCP directly — interacts with the host.

### 2. Host Application
Where the AI assistant runs: IDE, chat agent, copilot, enterprise workspace, desktop app.

**Owns:** user experience, workflow context, deciding when capabilities are needed, result integration.

**Should NOT own:** raw backend integrations, server-side business logic, capability definitions.

A good host is an orchestrator, not a monolith.

### 3. MCP Client
Protocol-aware component inside the host.

**Owns:** capability discovery, structured requests/responses, connection/session management, transport coordination.

**Should NOT own:** business workflows, data storage, custom logic belonging in server/backend layers.

Anti-pattern: embedding too much intelligence in the client.

### 4. MCP Server
Exposes capabilities through a standardized interface.

**Owns:** capability exposure, input validation, request routing, backend integration, response packaging, access controls.

The server is the abstraction layer in front of backend systems — not the system of record itself.

### 5. Tools
Executable, action-oriented capabilities.

Use when the assistant needs to **do** something. Produces a side effect or initiates one.

Examples: create a ticket, send a message, trigger deployment, update CRM record, run code action.

### 6. Resources
Retrievable content or data.

Use when the assistant needs to **read** something. Represents information access, not action execution.

Examples: fetch document, load config, retrieve logs, read knowledge base article, query dashboard.

Anti-pattern: turning every read operation into a tool. Weakens semantics and makes the capability surface less intuitive.

### 7. Prompts
Reusable prompt templates or guided interaction patterns.

Use to standardize how the host/assistant frames a task.

Examples: "Explain this code file for a junior engineer," "Generate postmortem summary from incident notes."

Prompts are structured reusable interaction assets, not execution endpoints.

### 8. External/Backend Systems
The real systems: APIs, databases, filesystems, repositories, SaaS, internal business systems.

Should remain behind the MCP capability layer. Exposing them directly creates tight coupling.

## Five-Layer Architecture

```
1. User Interaction Layer     → requests, questions, workflow initiation
2. AI Application Layer       → host + assistant, orchestration, context assembly
3. Protocol Layer             → MCP boundary, structured requests/responses, discovery
4. Capability Exposure Layer  → tools, resources, prompts modeled and exposed
5. Integration/Backend Layer  → real systems, execution, storage, system-of-record
```

## Separation of Concerns

| Layer | Owns |
|-------|------|
| **Host** | User interaction, assistant workflow, capability needs, result interpretation |
| **Client** | Protocol handling, discovery, request/response structure, transport |
| **Server** | Capability exposure, validation, backend integration, access control |
| **Backend** | Actual execution, persistent data, domain records, operational processes |

## Request Lifecycle

1. User sends request → host application
2. Host interprets intent
3. Client identifies relevant capability
4. Client sends structured request through MCP boundary
5. Server resolves to correct tool/resource/prompt
6. Backend interaction occurs
7. Server returns structured result (not raw backend complexity)
8. Host integrates result into user-facing response

## Deployment Shapes

| Pattern | Best For |
|---------|----------|
| Local Host + Local Server | Desktop assistants, privacy-sensitive workflows |
| Cloud Host + Remote Server | SaaS, shared services, global access |
| Enterprise Multi-Server | Domain separation, governance-heavy, large orgs |

## Common Mistakes

1. Mixing host logic and backend logic → tight coupling
2. Overloading server with UI behavior or presentation concerns
3. Treating everything as a tool — read paths should be resources
4. Exposing raw systems directly without abstraction
5. Weak protocol boundaries (inconsistent schemas, validation)
6. Unclear ownership of validation and auth
7. Designing for demo, not for scale

## Trade-Offs

**Benefits:** cleaner mental model, stronger modularity, better domain isolation, reusable capability surfaces, easier governance, long-term maintainability.

**Costs:** more layers, more upfront design, protocol modeling discipline, coordination overhead across servers, need for clear team ownership.

The trade-off is usually worth it once your system grows beyond a small prototype.

## Core Principle

> The host orchestrates, the client communicates, the server exposes capabilities, and the backend systems do the real work.

That mental model is the foundation for implementing, debugging, and scaling serious MCP-based systems.
