# How to Build AI Agent Swarms (Complete Guide)

This is a complete A–Z breakdown of AI Agent Swarms — what they are, how to use them, why they change everything about how you work with AI.

Kimi K2.6, Moonshot AI's April 2026 open-weight flagship, is the most serious open-source implementation of this idea.

Real tasks have width. Fifty companies to research. Two hundred files to analyze. A dozen subtasks that don't depend on each other and shouldn't wait in line. An agent swarm is the architecture for that.

This guide breaks down how it works, from the training infrastructure up to the API, then covers: Kimi for execution, Claude Opus 4.8 for planning and verification.

## Section 1: What is an agent swarm?

An agent swarm is multiple agents working simultaneously on decomposed subtasks, coordinated by an orchestrator that aggregates the results.

- **Sequential chain:** Agent A runs, hands off to B, B hands off to C. Total time = A + B + C.
- **Swarm:** Orchestrator splits the goal, agents A, B, C run at the same time. Total time ≈ max(A, B, C).

A swarm also solves context overflow. One agent on a long task accumulates tokens until its window drowns. A swarm gives each subtask its own bounded context.

## The six building blocks

| Component | What It Does |
|-----------|-------------|
| Orchestrator | Decomposes the task, assigns subtasks, monitors execution, aggregates results |
| Subagents | Specialized workers scoped to one domain (research, code, analysis, writing) |
| Tools | Functions agents can call: web search, code interpreter, file I/O, APIs |
| Memory | Shared state the swarm can read/write |
| Handoffs / Routing | Mechanism for passing control or data between agents |
| Guardrails | Iteration limits, timeouts, human-in-the-loop triggers, error recovery |

## Section 2: What Kimi K2.6 actually is

K2.6 is a 1-trillion-parameter Mixture-of-Experts model from Moonshot AI, released open-weight on April 20, 2026 under a Modified MIT License. Commercial use is free below $20M monthly revenue or 100M monthly active users.

**Architecture specs:**
- Total parameters: ~1.04 trillion
- Activated per token: ~32 billion (8 selected experts + 1 shared)
- Total experts: 384, across 61 transformer layers
- Context window: 262,144 tokens (262K)
- Attention: Multi-Head Latent Attention (MLA) — lower KV cache footprint
- Quantization: INT4 via QAT (~594GB on disk), FP16 ~2TB
- Runs natively on 4x H100 80GB (INT4), 8x H100 80GB (FP16)

## Section 3: The MuonClip optimizer

Training a trillion-parameter sparse MoE without it blowing up is hard. The specific failure mode: as sequence length grows, the query-key dot product can grow unbounded, producing loss spikes.

The Kimi K2 paper (arxiv: 2507.20534) introduces MuonClip: Muon optimizer (more token-efficient than AdamW) + QK-Clip (per-token, per-head clipping on QK matrices before softmax). This bounds attention score magnitude and kills explosion pathology.

Why this matters: a model trained with attention instability tends to hallucinate under long-context, high-step-count conditions — exactly the regime Agent Swarm lives in.

## Section 4: PARL — Parallel-Agent Reinforcement Learning

Agent Swarm is not a framework bolted on top of K2.6. The behavior was trained into the model via PARL (arxiv: 2602.02276).

**Trainable orchestrator, frozen subagents:** The orchestrator is trainable via RL on outcome rewards. Subagents are frozen intermediate policy checkpoints. Subagent trajectories are treated as environmental observations, not differentiable decision points.

**The three-part reward function:**
- Parallelism reward: pushes toward concurrent subagents
- Finish reward: ensures subagents actually complete tasks (blocks "spurious parallelism")
- Performance reward: scores final output quality

The optimization metric is **critical steps** (critical path length), not total steps — rewarded for shortening the longest dependency chain.

**Results:** BrowseComp 86.3% (K2.6 swarm), 3-4.5x wall-clock reduction on parallelizable tasks, up to 4,000 coordinated tool calls.

## Section 5: Mooncake serving infrastructure

Mooncake is Moonshot's KVCache-centric disaggregated architecture. Prefill and decode run on separate clusters. The KV cache spans GPU VRAM, CPU DRAM, and SSDs with a custom transfer engine.

When 300 sub-agents run simultaneously, Mooncake's disaggregated cache handles GPU memory pressure by evicting completed agent caches to DRAM/SSD.

From the paper: "Mooncake can achieve up to a 525% increase in throughput... operational across thousands of nodes, processing over 100 billion tokens daily."

## Section 6: How Agent Swarm works step by step

1. **Task decomposition:** Orchestrator analyzes the task and builds the dependency graph
2. **Specialist agent spawning:** Domain-specialized sub-agents (web research, data analysis, writing, fact-checker) with scoped tool access and bounded local context
3. **Parallel execution in waves:** Up to 300 sub-agents, 4,000 coordinated steps. Failed/stalled agents detected and reassigned automatically
4. **Aggregation and output:** Orchestrator synthesizes across agent outputs into final deliverable

## Section 7: The Kimi x Claude Opus 4.8 architecture

No single model is right for every layer. The pattern:

```
[Claude Opus 4.8 - Planner] → decomposes goal, identifies parallel vs sequential subtasks
         ↓
[Kimi K2.6 Agent Swarm - Executor] → spawns up to 300 specialized sub-agents, runs in parallel
         ↓
[Claude Opus 4.8 - Verifier] → reviews outputs against success criteria, flags gaps, synthesizes
```

Why Claude for planning/verification: Opus 4.8 is ~4x less likely than predecessor to let flaws pass unremarked. False confidence is the catastrophic failure mode in agentic systems. Also 1M token context window for pulling 50+ parallel agent outputs into one review context.

Why Kimi for execution: 300 parallel sub-agents, 4,000 tool steps, $0.95/$4.00 per million input/output tokens.

## Section 8: When you need a swarm (and when you don't)

**Stay single-agent when:** task fits in one context window (<50K tokens), sequential by nature, still prototyping, would complete in <10 minutes.

**Reach for swarm when:** n > 5 parallel independent subtasks, context overflow is real, need domain-specialized agents simultaneously, task too long for quality across one sequential session, want a critic/verifier checking another agent's work.

**Use Kimi + Claude hybrid when:** planning quality matters, output ships without human review, high-volume execution with cost concerns, want Claude's judgment on decision layers and Kimi's scale on work layers.

## Section 10: The four swarm architecture patterns

1. **Orchestrator-worker:** Central orchestrator → parallel workers → aggregator. Most common.
2. **Critic-refiner loop:** One agent produces, another critiques, repeat. Set max-iterations.
3. **Hierarchical:** Strategic orchestrator → domain orchestrators → workers.
4. **Claw Groups (Kimi-native):** Heterogeneous swarm coordinating any model (Claude, GPT, local models) + human workers.

## Section 13: The seven non-negotiable guardrails

1. Max iterations per agent
2. Session timeout
3. Structured output enforcement (JSON, not prose)
4. Failure isolation (sub-agent crash must not kill orchestrator)
5. Retry with exponential backoff
6. Human-in-the-loop checkpoints for write-access swarms
7. Cost monitoring — runaway loops show up as cost anomalies first

## Prompt design for swarm tasks

**Decomposition prompt:** "You are a task architect. Decompose this goal into independent, parallelizable subtasks. Output as JSON: {task_id, description, agent_type, depends_on, success_criteria}"

**Specialist prompt:** "You are a {ROLE} agent specializing in {DOMAIN}. Return ONLY valid JSON. If you cannot complete: {error, partial_results}"

**Aggregation prompt:** "Synthesize research from {n} agents. Identify agreement, disagreement, gaps. Call out inconsistencies explicitly — do not silently resolve contradictions."

## What to build first

Start with the three-agent pipeline (Plan → Execute → Verify). It's small enough to debug in an afternoon, exercises all three layers, and runs against a real task in under an hour. The failure mode will teach more about swarm design than another hour of reading.

The architecture is not the hard part. The hard part is the gap between "works in testing" and "works at 3am with nobody watching" — and that gap is entirely in the guardrails, the observability, and the memory design.
