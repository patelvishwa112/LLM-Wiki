# Your Agent Isn't Slow. It's Doing 4 Jobs at Once: A Field Guide to Agent Swarms That Don't Go Rogue

Nyk (@nyk_builderz)

You don't ship slower because Claude is dumb. You ship slower because one agent does 4 jobs. Research, code, review, orchestration — one context. By week 4 the roles blur and you lose the thread.

The real bottleneck isn't model IQ. It's not what you prompt, but how you wire it — the moat is architecture.

Most swarms that try this go rogue. Here's how I run them in production now, without the damage.

## Where a single agent stalls

A single agent reads, plans, acts, checks, plans again. For a narrow task, that's all you need.

It stalls when the job runs wide. "Find 50 companies, pull their filings, tell me which 5 are mispriced" is one sentence and 50 jobs.

Two things break. Time first: 50 independent parts in series cost about 50x, even though none waits on the others. Memory second: one context is one desk, and by company 40 it has lost what it found on company 3.

Then you become the glue — splitting work, pasting results between chats, tracking what's done.

A sharper model runs the same single line. Throughput is a shape problem, not an IQ problem.

## Two patterns people confuse

- **Sub-agents = compression.** Fan a wide task into parallel lanes, each a clean desk. Isolated sub-agents run backend work for about 1/3 the cost of a full team [E1].
- **Multi-agents = coordination.** Use these only when lanes must negotiate a shared decision, not only run in parallel.

Most teams over-engineer this. They reach for a committee when they needed compression.

## The org chart

- **Workers (sub-agents):** each gets its own memory, its own tools, one lane.
- **Coordinator:** splits the work, reads the plan, merges the results. Its only job is coordination.

## The part most swarm guides skip: governance

Parallelism gives you throughput. Throughput without governance is faster blast radius.

**PRODUCTION SWARM GOVERNANCE — four controls:**

1. **PRETOOL GUARD** (before any action) — e.g. rm -rf → trash; no push to main; no npm publish w/o approval; no curl | python; no chmod 777; no data exfil via curl

2. **HANDOFF ARTIFACT** (end of every session) — what changed, files touched, test status, git state, next steps → injected into the next run

3. **CONTROLLED DISAGREEMENT** (before converging) — force competing plans, score them, merge the winner

4. **COMPLETION GATE** (before the agent quits) — score session: completeness 35, honesty 30, deferral 20, work-evidence 15; can't exit by rationalizing "done"

The completion gate alone has scored 3,000+ sessions at 0 false positives [E2]. The disagreement step is a council of competing seats.

## How to run one

1. Decide if the job fits a swarm (wide + independent = yes; narrow + deep = single agent; don't fan if parts wait on each other)
2. Brief it like a team (role/goal, scope/guardrails, numbered deliverables, exact output format)
3. Hand over raw material in one pass
4. Read the coordinator's plan before it runs
5. Gate the output — score completeness, honesty, work evidence

## The takeaway

Change shape from line to grid; move from worker to orchestrator. Swarm buys throughput; governance prevents 2 a.m. cleanup.
