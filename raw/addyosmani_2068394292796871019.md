# How Modern Browsers Work (Chromium Architecture Deep Dive)

**Addy Osmani (@addyosmani)** posted a comprehensive technical overview (June 20, 2026) explaining the internals of modern web browsers, with a focus on Chromium.

## Overview

Web developers often treat the browser as a black box. In reality, a modern browser (Chrome/Chromium, Firefox/Gecko, Safari/WebKit) is a highly complex system orchestrating networking, parsing, rendering (GPU-accelerated), JavaScript execution, and security sandboxing across multiple processes.

The post provides a detailed walkthrough of Chromium’s architecture, while noting differences in other engines.

## Key Sections

**1. Networking and Resource Loading**
- Browser process handles navigation, URL parsing, security checks, DNS, TCP/TLS
- HTTP/2 & HTTP/3 multiplexing, MIME sniffing, CORB, Safe Browsing
- Speculative loading: DNS prefetch, preconnect, preload scanner, Early Hints (HTTP 103), Speculation Rules API, resource prioritization

**2. Parsing HTML, CSS, and JavaScript**
- Incremental HTML parsing → DOM construction (error-tolerant)
- Script blocking behavior (`async`/`defer`/`type="module"`)
- CSS parsing → CSSOM, style calculation, render-blocking resources

**3. Styling and Layout**
- Layout tree construction (reflow)
- Geometry calculation (box model, flex, grid, floats, etc.)
- Performance implications of layout thrashing

**4. Painting, Compositing, and GPU Rendering**
- Paint records / display list
- Layering and promotion (`will-change`, transforms)
- Compositor thread (separate from main thread)
- Tiling, rasterization (Skia), GPU textures
- Why transform/opacity animations are smooth (compositor-only)

**5. Inside the JavaScript Engine (V8)**
- Background parsing and compilation
- Pipeline: Ignition (bytecode) → Sparkplug (baseline JIT) → Maglev (mid-tier) → TurboFan (optimizing JIT, moving to Turboshaft)
- Type feedback, inline caches, deoptimization
- Orinoco garbage collector (generational, concurrent, incremental)
- Bytecode flushing
- Bindings to browser Web APIs

The post also covers module loading, multi-process architecture (browser process, renderer processes, GPU process, Network Service), site isolation, sandboxing, and DevTools performance insights.
