"Rahul\n@sairahul1\nSubscribe\nHow to Create Loops with Claude Code\n10\n15\n70\n41K\nMost Claude workflows are still one-shot.\nYou write a prompt. Claude responds. Session ends. Tomorrow you start from zero.\nThat's fine for one-off tasks.\nBut for repetitive work — daily reviews, CI triage, project monitoring, meeting follow-ups — you're rebuilding the same context every single day.\nThat's not a workflow. That's manual labor with AI sprinkles.\nThe better abstraction is a loop.\nSave this. You'll use it.\n\n\nWhat a Claude loop actually is\nA loop is not \"just ask Claude again.\"\nA loop is a repeatable workflow structure around the model.\nIt defines: \n→ What triggers the run \n→ What context Claude reads before acting \n→ What Claude is allowed to do → How the output is verified \n→ Where the state is stored \n→ When to stop, repeat, or ask for human review\nThe model still does the reasoning.\nThe loop provides the operating structure.\nThe shift matters because Claude sessions are temporary.\nWithout a loop, every run starts from zero. With a loop, Claude reads what happened before and continues from there.\n6 parts every Claude loop needs\nA loop without structure is just a prompt that runs twice.\nHere are the 6 parts that make a loop actually work:\n→ 1. Trigger\nWhat starts the run.\nManual command. Scheduled timer. File change. CI failure. Git event.\n→ 2. Context\nWhat Claude reads before acting.\nTask definition. Previous progress. Project files. Instructions.\n→ 3. Action\nWhat Claude does.\nWrite a report. Draft a fix. Classify failures. Update a document.\n→ 4. Verification\nHow you check the result.\nChecklist. Test suite. Required sections. Safety boundary check.\n→ 5. State update\nWhat gets remembered for next time.\nPROGRESS.md updated. Blockers recorded. Next run guided.\n→ 6. Decision\nStop. Repeat. Or escalate to human.\nThat's the full loop.\nAnd every step matters.\nSkip verification → the loop trusts itself blindly.\nSkip state → every run starts from zero.\nSkip the decision → the loop never knows when to stop.\nThe 4-file system that makes loops work\nYou don't need infrastructure.\nYou need 4 files and a folder.\nplaintext\nmy-loop/\n├── TASK.md\n├── LOOP_INSTRUCTIONS.md\n├── PROGRESS.md\n└── outputs/\n    └── daily-review.md\nTASK.md — The goal. What this loop is trying to accomplish.\nLOOP_INSTRUCTIONS.md — The operating procedure. How Claude should run, what it's allowed to do, how to verify.\nPROGRESS.md — The memory. What happened last time, what's blocked, what the next run should focus on.\noutputs/ — Where Claude writes results. Predictable. Inspectable. Safe.\nThat's it.\nClaude reads these files at the start of every run. Updates them at the end. Next run continues from where the last one stopped.\nThis is what makes loops different from repeated prompts.\nState lives outside the chat.\nTASK.md — Define the goal clearly\nThis file tells Claude what the loop is for.\nKeep it high-level. Don't put operating details here. Those go in LOOP_INSTRUCTIONS.md.\nCopy this template:\nmarkdown\n# Daily Project Review Loop\n\n## Goal\nReview this project folder, summarize what changed,\nidentify blockers, and produce a short daily review report.\n\n## Expected Output\nEach run should produce or update:\n- `outputs/daily-review.md`\n- `PROGRESS.md`\n\n## Scope\nClaude may inspect files in this workspace and write\nreports to the `outputs/` folder.\n\nClaude should NOT:\n- Modify source files\n- Delete or rename files\n- Send messages or open tickets\n\nNotice the last section.\n\nThe first version of any loop should have explicit \"do not\" rules.\nThis is not Claude being untrusted.\nThis is you being a good engineer.\nPROGRESS.md — The loop's memory\nThis is the most important file in the system.\nWithout it, every run starts from scratch. With it, Claude knows what happened and continues from there.\nThe structure that actually works:\nmarkdown\n# Loop Progress\n\n## Current State\n- Status: Active\n- Main objective: Daily project review\n- Current focus: [what matters right now]\n- Last updated: [date]\n\n## Last Run\n- Date:\n- Summary:\n- Files reviewed:\n- Output produced:\n\n## Open Items\n-\n\n## Blockers\n-\n\n## Needs Human Review\n-\n\n## Next Run Should\n-\n\n## Decisions Made\n-\n\n## Do Not Repeat\n-\nThe \"Do Not Repeat\" section is critical.\nIt prevents Claude from retrying things that already failed.\nThe \"Needs Human Review\" section is critical.\nIt stops the loop from silently continuing when it shouldn't.\nTwo rules for keeping PROGRESS.md useful:\n→ If Claude needs it to decide the next action → keep it in PROGRESS.md → If you just want a record → move it to outputs/history/\nState files that grow forever become useless.\nKeep PROGRESS.md as a control panel, not an archive.\nLOOP_INSTRUCTIONS.md — The operating procedure\nThis file is the control layer around Claude.\nIt tells Claude exactly what to read, what to write, what not to touch, and how to verify the result before stopping.\nThe template that works:\nmarkdown\n# Loop Instructions\n\nYou are running a daily project review loop.\n\n## Before You Start\n1. Read `TASK.md`\n2. Read `PROGRESS.md`\n3. Inspect the project folder\n4. Identify what changed, what's incomplete, what needs human review\n\n## What You Should Do\nWrite a short daily review to `outputs/daily-review.md` including:\n- Summary of current state\n- Files reviewed\n- Meaningful changes\n- Blockers or unresolved questions\n- Recommended next actions\n\nAfter writing, update `PROGRESS.md` with:\n- Date of this run\n- Summary of what happened\n- Files checked\n- What the next run should do\n- Whether human review is needed\n\n## Safety Rules\n- Do not delete files\n- Do not rename or move files\n- Do not modify source files\n- Only write to `outputs/daily-review.md` and `PROGRESS.md`\n- If unsure whether an action is allowed → stop and ask\n\n## Verification Checklist\nBefore ending the run, confirm:\n- `outputs/daily-review.md` exists with all required sections\n- `PROGRESS.md` was updated\n- No files outside allowed paths were modified\n\n## Failure Policy\nIf verification fails:\n1.
