# AutoData: Synthetic data generation explained

**Author:** AVB (@neural_avb)  
**Paper:** http://arxiv.org/abs/2606.25996  
**Paper Breakdown:** https://paperbreakdown.com/abs/2606.25996  
**Posted:** Jul 1, 2026

This new paper Autodata tries to automating the creation of high-quality synthetic datasets using an agentic "data scientist" rather than a single prompt or one-shot generator.

Data quality is the SINGLE MOST important thing when training small language models (and often big ones too). We need useful, correct, diverse, and aligned with the downstream goal for the model.

Co-authored by AVB and Claude Sonnet inside the Paper Breakdown harness.

## "Synthetic data" is cheap but "good synthetic data" is not

Generating synthetic data is easy (prompt an LLM to "write 10,000 examples"). That usually fails: incorrect, too easy, repetitive, or shortcut-exploitable examples. Quality requires iteration: propose, check, revise prompts, select good items.

Autodata: an LLM agent produces a dataset optimizing downstream performance:

- **Input:** target task/domain + inference-time compute budget
- **Process:** agentic loop — generate candidates, challenge/verify, select/refine
- **Output:** synthetic train/eval data that improves model performance

## Goal: better reasoners via GRPO (not SFT)

Target: CS research, legal analysis, math/scientific reasoning. Training method: **GRPO** (Group Relative Policy Optimization), not SFT.

- SFT: (input, correct output) imitation
- GRPO: multiple rollouts per question, rewards via rubric/verifier, advantage from within-group comparison

## General Autodata loop

1. **Data Creation:** agent grounds on source material, uses tools + compute for candidate examples
2. **Data Analysis:** per-example correctness/difficulty/quality; dataset-level diversity and training value
3. **Iterate / Stop:** feedback refines creation until quality threshold; guardrails prevent hacking quality checks

Outer **meta-optimization** can tune the agent to become a better data scientist.

## Agentic Self-Instruct (concrete algorithm)

Orchestrator delegates to four subagents: **Challenger**, **Weak solver**, **Strong solver**, **Judge**.

Experiment stack: Kimi-K2.6 (challenger + judge), Qwen3.5-4B (weak solver), Qwen3.5-397B-A17B (strong solver); GRPO fine-tune weak solver on accepted data.

Steps:

1. Grab source corpora (papers, legal docs, medical articles, etc.)
2. **Propose:** Challenger generates candidate from grounded context
3. **Stress-test:** weak + strong solvers attempt the example
4. **Judge:** evaluates outputs and example quality (well-posed question, correct rubric)
5. **Acceptance:**
   - *Verifiable tasks:* strong majority correct, weak majority wrong (difficulty gap + correctness)
   - *Non-verifiable:* judge quality gap — not too easy/hard for weak solver; strong solver confirms via rubric
6. **Refine or Accept:** if fail, orchestrator revises Challenger prompt from judge report; if pass, add to dataset

Weak/strong can be same base model in different modes (more compute, scaffolding, privileged info).

## Quality enforcement (Judge)

- Correctness / leakage: reject if answer constructible from context without reasoning
- Reasoning vs recall: flag recall-only ("what/which/how many") vs reasoning ("why/what-if/predict/decide")
- Rubric structure (non-verifiable): 10–15 criteria, ≥4 positive / ≥3 negative, criteria require reasoning beyond context
- **Weak/strong gap:** weak avg ≤65% (no high individual runs), strong ≥60% and <95%, gap ≥20% — proxy for useful GRPO gradient (avoids all-zero or all-hundred reward groups)

## Diversity / mode collapse

- Challenger told each refinement round to use "ENTIRELY NEW question from DIFFERENT angle"
- Grounding on varied real documents (different paper/doc per example)
- Challenger reads source file directly before generation

## Meta optimization / auto-research

Outer loop optimizes agent prompts/strategy using same weak/strong gap as fitness:

- Population of prompt diffs vs baseline repo
- Boltzmann parent selection
- Evaluate parent on minibatch of papers (full inner loop + trajectories)
- Analyzer LLM diagnoses failure patterns
- Implementer mutates prompt → new diff
- Re-evaluate parent + mutant on held-out validation papers
- Mutant kept only if validation score strictly beats parent
- Parallel iterations across population

## Evaluation

1.3k–2.8k examples per source; GRPO train Qwen3.5-4B (batch 16, lr 1e-6); held-out test eval.

**CS example:** 10k+ S2ORC papers (2022+). Challenger outputs: question type, 2–3 reasoning-skill tags, non-leaking context, deep-reasoning question, reference answer, weighted 10–15 criterion rubric. Accept when strong avg ≥0.65, weak avg <0.5.

**Result:** Agentic Self-Instruct dataset beats CoT Self-Instruct baseline on held-out tests; other domains in paper appendix.
