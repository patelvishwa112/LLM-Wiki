# Production AI Systems: RAG and Agents (Part 2)

**Post by @techNmak**

This is a high-quality post focusing on the two layers that surround the model in real production systems: **RAG** and **Agents**. (Part 2 of a series; Part 1 covered the model layer.)

### Core Thesis
Production AI systems rarely fail because the model doesn't understand self-attention. They fail at **retrieval** (the right evidence never reaches the model) and **agent design** (tools, observation, planning, and verification are fragile).

### RAG – Key Insights
RAG is not "just put documents in a vector DB." It's a complex pipeline where every stage can silently break the system:

- **Ingestion & Parsing**: Quality starts here. Bad parsing of tables, headings, hierarchies, figures, or PDFs destroys everything downstream.
- **Chunking**: Not just size. Structural (headings, sections, code blocks) and semantic chunking are usually superior to fixed-size. Strategy must match document type and query patterns.
- **Retrieval**: Hybrid (dense + sparse + metadata filters) beats pure vector search in most real corpora. Exact terms, IDs, dates, and legal language need sparse/keyword support.
- **Re-ranking, Metadata, Freshness, Versioning, Access Control**: These are non-negotiable in enterprise settings. Permissions must be enforced *before* content reaches the model.
- **Evaluation**: Separate retrieval metrics (recall, NDCG, context precision/recall) from generation metrics (faithfulness, groundedness, answer relevance). Many "generation" failures are actually retrieval failures.
- **Multimodal RAG**: Evidence is often visual (charts, diagrams, regions of pages). Text-only retrieval is insufficient.
- **Other production realities**: Embedding model migration at scale, prompt injection defense (treat retrieved content as untrusted), proper grounding/citation evaluation.

### Agents – Key Insights
An agent isn't just "LLM that uses tools." It's a system that pursues goals, observes results, and replans.

- Prefer **pipelines** unless dynamic decision-making is genuinely required. Agents add complexity, cost, and failure modes.
- Core principle: **Model proposes. System verifies. Tool executes.**
- **Tool design** is critical. Broad tools (`run_command(input: string)`) are dangerous. Narrow, schema-validated, permission-checked, idempotent tools with clear success/failure signals are much safer.
- **Observation design** matters as much as the tool itself. Ambiguous tool responses cause the agent to misinterpret results.
- Strong termination conditions, loop detection, step/budget limits, and programmatic success checks are required.
- **Memory** (working, episodic, semantic, procedural) should be deliberate — not accumulated by default. Consider staleness, privacy, and retrieval quality.
- Multi-agent systems add coordination overhead and often create more problems than they solve unless there's a clear reason (specialist roles, permissions, parallel work).
