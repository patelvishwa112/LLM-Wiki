# From 1 agent to a swarm: a roadmap for orchestrating agent loops

**Author:** Morlex (@0xMorlex)  
**Source:** https://x.com/0xmorlex/article/2070079645148451263

One agent is a craftsman. It works on one thing, in one context, and you can read everything it did. A swarm is a small organization: many agents working at once, and your job stops being the work and becomes the coordination. That shift is where most people either unlock real throughput or create an expensive mess that fails in ten places at once.

The mess is the default outcome, because a swarm multiplies whatever you point it at. Point ten agents at an unclear goal and you get ten flavors of wrong, in parallel, colliding on the same files.

The skill is not spawning more agents. It is orchestrating them: giving them lanes, a shared goal, a way to not step on each other, and a check that decides when the whole thing is done.

This is the roadmap from one agent to a swarm. Three steps to decide when and how to split, four to run the swarm without it eating itself, three to keep it sane at scale. Every part uses Claude Code primitives you already have, and everything factual is checked against the current docs.

## Tier 1 · Decide to swarm

**01. Don't swarm by default.**  
The first move is restraint. A single agent in a tight loop handles most work, and it is far easier to debug than five agents talking past each other. Reach for a swarm only when one agent is genuinely the bottleneck: the work is embarrassingly parallel (forty files to migrate, thirty tests to triage), or it needs several independent perspectives that pollute each other in one context.

If the task is sequential, or small, or you cannot yet describe "done" precisely, you are not ready to swarm. Get one agent reliable first. A swarm built on a loop you have not stabilized just produces failures faster and in more places.

**02. The shape: one orchestrator, many workers.**  
A swarm is not a pile of equal agents. It is a hierarchy: one orchestrator that holds the plan and the goal, and many workers that each do a narrow slice. The orchestrator does not write the code. It decomposes the job, hands each piece to a worker, collects results, and decides what happens next.

This is the pattern that scales, because the orchestrator's context stays clean. It never sees the forty files a worker read or the wall of logs a test produced. It sees the plan and the verdicts. Keep the lead's job to thinking and routing, and push all the noisy doing down to workers.

**03. The unit: a subagent with its own context.**  
The worker is a subagent: an independent Claude session with its own context window and its own tool list. That isolation is the whole point. Each worker reads only what its task needs, and its mess never reaches the orchestrator or the other workers.

A good worker has a narrow brief and stays in its lane (example frontmatter: name fixer, tools Read/Edit/Bash, single failing test only). Define workers with /agents, give each the minimum tools it needs, and resist the urge to make one do-everything agent. Narrow workers are easier to verify and easier to run in parallel.

## Tier 2 · Run the swarm

**04. Parallelism without collisions.**  
Git worktrees: each worker gets a separate working directory on its own branch. Claude Code can also isolate a subagent to its own worktree. Without this, parallelism is not speed, it is a merge conflict waiting to happen.

**05. Put an independent verifier in every lane.**  
At swarm scale, self-review fails quietly and at volume. Every worker's output goes to a separate verifier with a fresh context window. Adversarial verification is the single highest-leverage role in a swarm. The orchestrator only merges what an independent check passed.

**06. Compose the shapes with workflows.**  
Have Claude write the orchestration itself — fan out per failing test, verifier checks every diff before merge. Three shapes: fan out and synthesize, adversarial verification, loop until stop condition.

**07. One goal, one stop condition.**  
Give the whole swarm a single objective with /goal that an independent grader checks. The goal belongs to the orchestrator, not the workers.

## Tier 3 · Keep the swarm sane

**08. Give the swarm a shared memory.**  
Shared state file: workers claim a task before starting and record when done (queue / in progress / done).

**09. Route by cost.**  
Heavy model for orchestrator, cheap for workers, cheapest for graders.

**10. You cannot watch ten agents. Build the guardrails.**  
Structural safety: permissions deny list, hooks (e.g. PostToolUse logging). Deny the irreversible, log the rest.

## Mistakes that turn a swarm into a pileup

- Swarming before one agent is reliable  
- Equal agents, no orchestrator  
- Parallel edits without worktrees  
- Workers grading themselves  
- No shared memory  
- One model for everyone  
- Supervisory safety instead of structural guardrails  

## The point

A swarm is not a smarter agent. It is the same agents arranged into an organization: one lead, narrow workers, independent checks, shared goal, shared memory, cost routing, hard walls. Get one agent reliable first; add verifier, worktree isolation, or shared state before spawning the next agent. Leverage is orchestration, not headcount.
