# Claude Fable 5 — map vs territory and unknowns

**Author:** Thariq (@trq212)  
**Posted:** 2026-07-03  
**Engagement:** 2,001 likes, 159 reposts

Working with Claude Fable 5 keeps re-teaching: **the map is not the territory.**

- **Map:** prompts, skills, context you give Claude  
- **Territory:** codebase, real world, actual constraints  
- **Unknowns:** gap between map and territory—Claude guesses what you want when it hits them  

With Fable, work quality is **bottlenecked by clarifying unknowns** (before, during, after implementation).

## Four buckets of unknowns

- **Known knowns** — what's in the prompt  
- **Known unknowns** — gaps you're aware of  
- **Unknown knowns** — tacit criteria you'd recognize when seen  
- **Unknown unknowns** — blind spots  

Reducing/planning unknowns = core **agentic coding** skill.

## Balance when instructing

Too specific → Claude won't pivot; too vague → generic best practices misaligned with your task.

## Techniques

**Pre-implementation**
- **Blind spot pass** — "unknown unknowns" in unfamiliar domains  
- **Brainstorms/prototypes** — HTML artifacts, fake data, multiple design directions  
- **Interviews** — one question at a time, architecture-changing ambiguities first  
- **References** — source code as best reference (Claude Design reads site code)  
- **Implementation plans** — lead with tweakable decisions (data model, UX), bury mechanical refactors  

**During implementation**
- **implementation-notes.md** — log deviations at edge cases, conservative choices, keep going  

**Post implementation**
- **Pitches/explainers** — package demo + spec + notes for buy-in  
- **Quizzes** — HTML report + quiz before merge
