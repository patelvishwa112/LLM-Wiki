# How to Build Your Own Agent Harness

By Mike Piccolo (@mfpiccolo) | Founder & CEO @iiidevs | May 28, 2026

## Core Argument

Most agent teams don't build a harness. They adopt one — LangChain, LangGraph, OpenAI Agents SDK, CrewAI, etc. This is wrong. Every long-running agent team eventually rewrites its harness because the framework bundles 10-12 different things together and you can't swap one without forking the whole thing.

iii's bet: a harness is a set of independent workers on a shared engine, each replaceable, each versioned independently, connected by a single primitive (iii.trigger()). The harness becomes a stack of installable workers. "Build your own" means "swap a few workers."

## The 15 Jobs a Harness Has to Do

1. Accept a turn request and persist it
2. Resolve credentials for model providers
3. Look up model capabilities (vision, tools, streaming, context window)
4. Drive the per-turn state machine
5. Load and serve skill bodies for function descriptions
6. Assemble the system prompt (mode, identity, skills appendix)
7. Stream tokens back to the client
8. Check every tool call against policy before execution
9. Pause tool calls needing human approval, route answers back
10. Track LLM spend against budgets
11. Run hooks before/after tool calls (logging, redaction)
12. Persist session as branching tree for forks/resumes
13. Compact session history when context fills up
14. Emit event stream for UI subscriptions
15. Carry OpenTelemetry trace across every step

## The iii Stack (14 Workers)

Each worker speaks the same WebSocket protocol, registers functions/triggers on the engine bus, and is independently swappable. The production stack:

- **harness/meta** — trigger router, OpenTelemetry seed, policy engine
- **turn-orchestrator** — durable per-session FSM
- **auth-credentials** — file-backed credential resolution
- **models-catalog** — static model capability catalogue
- **provider-anthropic** — Anthropic SSE streaming
- **iii-directory** — skill resolution and on-demand body fetching
- **approval-gate** — human-in-the-loop tool call parking
- **llm-budget** — per-workspace spend tracking
- **hook-fanout** — before/after tool call hook dispatch
- **session** — branching session tree persistence
- **context-compaction** — turn-boundary compaction
- **agent-events** — event stream fanout for UI
- **turn-sandbox** — per-run iii-sandbox microVM lifecycle
- **harness-integration-tests** — cross-worker integration tests

## Key Design Decisions

- **Thin vs thick is a slider, not a choice.** A thin harness is turn-orchestrator + provider + auth. A thick harness is all 14 workers. Same wire protocol. You move the slider by adding/removing workers in config.yaml.

- **Fail-closed by construction.** If the policy worker is unreachable (5s timeout), consultBefore denies the call. No silent approvals.

- **Approval wake is reactive and shared.** One state trigger (turn::on_approval) covers every session. No per-call resume functions.

- **Major refactors stay contained.** The turn-orchestrator collapsed from 11 FSM states to 7. Zero changes needed in any other worker. The contracts (function IDs on the bus) held.

- **Every worker is built on the same primitive your business logic uses.** The harness isn't special. It's just workers.

## Five Swappable Examples

1. **Replace model catalogue** — write a worker registering models::list/get/supports backed by live API. Swap it in.
2. **Add a new provider** — one folder, one iii.worker.yaml, one register.ts. Available instantly.
3. **Serve skills from private artifact store** — register directory::skills::get/list backed by S3/docs. Disconnect default.
4. **Override system prompt** — pass system_prompt in run::start. Orchestrator uses it verbatim.
5. **Replace approval UI** — write a Slack worker calling approval::resolve. The orchestrator never knows.
