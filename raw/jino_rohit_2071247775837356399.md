## What is even a FX graph?

What's a FX graph? Why should I care? I want to read torch.compile and skip this boring graphs.

Bad news! understanding fx graphs are fundamental to understanding every other components of the compile ecosystem. From dynamo to inductor, each of them make use of fx graph in some way or the other.

Good news! Im going to unpack everything in one blog!

**FX Graph** is an intermediate representation (IR) provided by a PyTorch module that transforms Python code into a directed acyclic graph (DAG).

### The good ol' DAGs

DAGs stands for Directed Acyclic Graph that is basically made up on nodes and directed edges.

### The Three Core Objects

FX Graph has exactly three concepts you need to understand — **Graph**, **Node**, and **GraphModule**.

Example traced with `torch.fx.symbolic_trace`:

```python
import torch
import torch.fx

class MyModel(torch.nn.Module):
    def __init__(self):
        super().__init__()
        self.linear = torch.nn.Linear(4, 8)

    def forward(self, x):
        return torch.relu(self.linear(x))

gm = torch.fx.symbolic_trace(MyModel())
gm.graph.print_tabular()
```

Tabular output:

```
opcode         name    target                 args         kwargs
-------------  ------  ---------------------  -----------  --------
placeholder    x       x                      ()           {}
call_module    linear  linear                 (x,)         {}
call_function  relu    <built-in fn relu>     (linear,)    {}
output         output  output                 (relu,)      {}
```

### Graph

Textual representation:

```
graph():
    %x : [num_users=1] = placeholder[target=x]
    %linear : [num_users=1] = call_module[target=linear](args = (%x,), kwargs = {})
    %relu : [num_users=1] = call_function[target=torch.relu](args = (%linear,), kwargs = {})
    return relu
```

### Node

Six fields: `op`, `name`, `args`, `kwargs`, `users`, `target`, `meta`.

**node.op** — six values: `placeholder`, `get_attr`, `call_function`, `call_method`, `call_module`, `output`.

**target** meaning depends on `op` (e.g. `call_module` vs `get_attr`).

**users** — use-count for graph transforms and dead-code detection.

## How FX Graphs are Built — Symbolic Tracing

Normal execution loses the computation graph. Symbolic tracing intercepts ops via **Proxy** objects (`__add__`, `__torch_function__`, `__call__`, etc.) and records **Nodes** without running real math.

### Limitations of Symbolic Tracing

1. **Dynamic control flow** — `if x.sum() > 0` cannot be evaluated on Proxies.
2. **Non-torch Python functions** — e.g. `math.sqrt` not intercepted.
3. **Static control flow** — `if self.do_activation` in `__init__`-known flags traces fine.
