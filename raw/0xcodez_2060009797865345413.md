# How to Master Claude Code: 10 Techniques

By Codez (@0xCodez) | May 28, 2026 | 53K views | 122 bookmarks

4% of all public GitHub commits are now written by Claude Code (~135K/day). Most people get a third of what it can do. The fix is not a better model — it's a better setup.

## PART 1 · Foundation

### 01. Write a CLAUDE.md
Read into context at the start of every session. For things that must be true on every turn: stack, conventions, non-negotiables. Cascades: project root (team, via git), ~/.claude/ (personal, all projects), nested (subfolder rules). Keep it lean — every line costs context.

### 02. Plan before you code
Ask for the approach, files, tradeoffs, assumptions — and approve before code. A one-line plan fix costs seconds; a wrong implementation across 40 files costs context and time. Add "don't write code yet" explicitly.

### 03. Scope tightly
"Add a CSV-export button to the reports page that exports filtered rows" — clear edge. "Improve the reports page" invites chaos. If you catch yourself writing "and" in a request, that's two tasks.

## PART 2 · Control

### 04. Keep diffs small and reviewable
A bounded task produces a diff you can read in under a minute. Code you don't review is code you don't understand — when it breaks at 2am, you're debugging a stranger's work.

### 05. Make Claude run the tests
Not "did you test it?" but "run the suite and paste the result." Test-first is stronger: write the test, watch it fail, then implement until green. A failing-then-passing test proves it exercises the new behavior.

### 06. Use hooks for what must always happen
CLAUDE.md is advisory ("usually"). Hooks are deterministic ("always"): PreToolUse (block dangerous commands), PostToolUse (format after every edit), UserPromptSubmit (inject context). Shell scripts that fire automatically and cannot be skipped.

## PART 3 · Scale

### 07. Connect MCP servers
MCP = USB-C for AI. Connect databases, issue trackers, design tools. MCP is for access, Skills are for know-how. Curate deliberately — a 5-server setup costs ~55K tokens upfront vs ~100 tokens for a dormant Skill.

### 08. Write /Skills for repeated workflows
SKILL.md with YAML frontmatter + markdown instructions. Progressive disclosure: only name/description (~100 tokens) at session start, full body only when task matches. If you've written the same instructions twice, it should be a Skill.

### 09. Delegate to subagents
Spawn for research, code review, exploring unfamiliar code. Fresh isolated context, own tools, own model (cheap Haiku for grep-heavy work). Clean summary back; noise never touches main session.

### 10. Match the model to the task
Opus for architecture/gnarly debugging. Sonnet for daily building. Haiku for fast mechanical work ("find every file importing X"). Switch mid-session. Make it a choice, not a default.
