# Continuous batching for GRPO rollouts in TRL

**Sergio Paniego** (Machine Learning Engineer at Hugging Face) posted a detailed technical update on bringing continuous batching into training for GRPO in TRL.

## Key Points

- Provides an efficient **in-process generation path** for rollouts in online RL (GRPO / RLOO) without relying on vLLM or the default wasteful `generate()`.
- Activated with a single flag: `use_continuous_batching=True` on `GRPOTrainer` / `RLOOTrainer`.
- Especially effective at high N (≥32 completions), particularly with variable-length generations (e.g., math reasoning).
  - At N=64: ~1.25x faster **and** lower VRAM usage than default `generate()` (recycles KV cache slots instead of eagerly allocating for max length).
- `max_memory_percent` defaults to 0.5 (leaves headroom for backward pass; lower than transformers default).
- CUDA graphs are disabled (weights change every training step).
- Fixes a prior limitation: `use_transformers_paged` silently dropped logprobs (breaking importance sampling correction). The new continuous batching path properly captures them.
- Requires `transformers>=5.8.0`. Currently available on main; will ship in the next TRL release.
- Includes links to:
  - PR #5765
  - Example script
  - Benchmark gist
  - Blog posts explaining how continuous batching works

This is a significant efficiency improvement for agent/RL training infrastructure at scale, directly relevant to rollout optimization, agent traces/trajectories, and high-throughput trainer loops.
