# How do you write a good skill? There's actual data now.

**Co-authored:** @seldo  
**Posted:** Jul 7, 2026 · AIEWF follow-up (skills theme)

Skills were a top theme at AI Engineer World's Fair in San Francisco last week. Latent Space's AINews said so, Paul Bakaus ran a workshop on the dark art of building skills, and vendors from Anthropic to GitHub are now shipping official skills as product surface. But for most AI engineers, writing a good skill has been folklore: talks, tweets, and trial and error.

That can now change. Three papers, one published in February and two in June, measured skills the way we measure everything else in this industry: run the tasks, score the results, compare the conditions. The result is a real rulebook for writing skills, with a bunch of counter-intuitive advice.

## What counts as a skill?

A skill is a folder of packaged domain knowledge that an agent loads on demand: instructions, vocabulary, sometimes scripts and resources. Spectrum: prompts → tools → **recipe** (Roland Gavrilescu / Introspection): larger container adding evals, judges, feedback signals.

**SkillsBench** (arXiv:2602.12670): modular, reusable, procedural, portable across models. Analogy: models = CPUs, harnesses = OS, skills = applications.

## Five key findings

### 1. Don't ask the model to write its own skill

SkillsBench: no skills vs expert-curated vs self-generated. Curated raised pass rates; self-generated averaged **1.3 points below** no-skill baseline. Models cannot reliably author procedural knowledge they consume. Failure modes: vague procedures or not recognizing specialized knowledge needed. SkillComposer (arXiv:2606.06079) uncontrolled log extraction hurt coding below baseline.

Paul Bakaus / Impeccable: precise professional vocabulary (e.g. "bold" via hierarchy/typography). Gavrilescu: cannot capture all knowledge in Markdown without human extraction.

### 2. Writing an exhaustive skill is not the way

Focused **2–3 file** skills beat large bundles and encyclopedic docs; comprehensive skills **lowered** pass rates below no-skill. Encode judgments the model repeatedly gets wrong, not full manuals.

### 3. Loading more skills makes agents worse

SkillComposer (arXiv:2606.32025): full **196-skill** library **−16 points** vs small relevant subset, **+23%** tokens. Tiny 3.9M-param composer for skill ID ordering +23.1 points on GPT-5.2-Codex. Routing (MoE-style) inside or outside skill matters.

### 4. Test in every harness you claim to support

Same skills: gains **4.1–25.7 points** by harness/model. Claude Code highest utilization; Codex CLI often ignored skills after acknowledging them. Per-harness compiled builds (Impeccable). Portability is engineering per target.

### 5. Aim skills where the model is weakest

Gains: **+4.5** software engineering vs **+51.9** healthcare; manufacturing high. **38%** of public skills are SW dev — crowded low-edge domain. Arbitrage: internal conventions, regulated industries, thin pretraining.

## You can't know your skill works without testing

Run tasks with/without skill, automated pass/fail scoring. SkillsBench: **16/84** tasks worse with skills loaded — outputs can look more professional while failing more. Skill evolution: accept changes only if pass rate rises meaningfully. Gaps: subjective domains, homogenization risk (Bakaus: same design skill → same-looking UIs).

## Checklist (folklore era over)

- Write it yourself
- Keep it small and focused
- Don't load too many
- Test per harness
- Target weak domains
- Verify every change with evals

**Papers:** SkillsBench (2602.12670), SkillComposer evolution (2606.06079), SkillComposer composition (2606.32025).
