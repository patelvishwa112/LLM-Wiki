# Your Agent Harness Should Repair Itself

**Author:** @akshay_pachaar (Akshay)
**Source:** https://x.com/akshay_pachaar/status/2064051835636498924
**Date:** 2026-06-08

---

When an AI agent fails in production, your observability tool shows you exactly what it did and almost nothing about how to fix it.

You get a clean trace of the run, every model call and tool that fired, how long each step took, and what it cost in tokens. What you don't get is why it broke, the change that would fix it, or any promise the same thing won't happen again next week.

So you scroll through the trace span by span, form a theory about what went wrong, write a patch by hand, and hope it doesn't break something that was working before. Then a new model ships with a fresh batch of failure modes, and you run that whole manual loop from the top.

The real bottleneck isn't your observability. It's everything that has to happen after the trace lands on your screen.

## Why Current Observability Breaks at Scale

Most agent observability platforms deliver a trace and stop:
- "What happened" → the platform handles this
- "Why it happened" → manual
- "Here's the fix" → manual
- "This won't break again" → manual

That was reasonable in 2023. It's the wrong abstraction for teams running agents in production today. Every model upgrade introduces new failure modes. Every new tool adds new edge cases. The harness gets more complex faster than any team can manually track and repair.

## Opik: The Four-Layer Stack

Opik is an open-source logging, debugging, and optimization platform for AI agents, built around the premise that this loop should be automated, not staffed.

### Layer 1: Tracing
Every LLM call, tool invocation, and retrieval step instrumented automatically with @opik.track. Works with LangGraph, CrewAI, and 50+ frameworks.

### Layer 2: Ollie
A coding agent built into Opik. Reads span trees, identifies failure modes, explains causal chains across LLM calls. With opik connect: reads source files, identifies exact lines responsible, proposes a diff (requires explicit approval). After approval: reruns agent against original failing inputs, streams new trace for side-by-side comparison, locks original failure as regression test.

### Layer 3: Test Suites
Plain-English assertions instead of numerical metrics. Every failing trace you debug automatically becomes a new test case. The suite grows from real production failures, not synthetic scenarios.

### Layer 4: Agent Sandbox
Runs the fully instrumented agent end-to-end inside the UI. Change a prompt, swap a model, add a tool, watch how the whole system responds. Non-developer stakeholders can test configurations without touching git.

## The Flywheel
Instrument → something fails → Ollie reads trace and source → proposes fix → you approve → rerun in Sandbox → passes → locked as regression test → next failure enters same loop. Every cycle, the harness gets harder to break.
