# How I use AI to find PRD flaws before review

George from prodmgmt.world (@nurijanian)

**Thesis:** Docs get judged by people who weren't in the room when you wrote them. "Socialize early" fails because careful reading happens only when a meeting is on the calendar — then you defend instead of improve.

**Move (10 words):** See your doc through every reviewer's eyes before sending.

**Mechanism:** Fan out agents (Claude Code dynamic workflows, May 28 — up to ~1000 parallel agents). Each agent inhabits one reviewer lens:
- Engineering
- Design
- Executive
- Legal and risk
- UX research
- Customer
- Devil's advocate (weakest claim only)

All read in parallel; objections return before the doc leaves the author.

## Example: in-app notification PRD (seven lenses, one pass)

- Engineering: "real-time delivery" undefined; assumed unscoped websocket layer
- Design: no empty / zero-notification state
- Executive: reads like a feature not a bet — what goal does it move?
- Legal/risk: push payloads may carry personal data → compliance
- UX research: no opt-out granularity (all-or-nothing)
- Customer: assumes users want *more* notifications
- Devil's advocate: "drives engagement" — no baseline or number

Author: three would have landed live in exec review; fixed privately first. ~Week of async comments → one pass, zero meetings.

## Alternatives (thinner, not repeatable every Tuesday)

- One trusted senior PM (one lens, social capital cost)
- Seven pre-meeting 1:1s (full coverage, calendar + staleness)

## PM OS workflow name

Repeatable fan-out packaged as **multi-perspective-review** — seven named reviewer agents, same lenses every PRD.
