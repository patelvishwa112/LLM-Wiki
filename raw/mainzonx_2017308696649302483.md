## From Single Chips to Coherent Clusters

In Part 1, we viewed the GPU as a self-contained machine. For frontier models, a single chip is no longer a machine. It is merely a component.

The shift to multi-GPU systems is driven by a hard physical reality: capacity. A 405B parameter model in FP16 requires over 800GB of VRAM just to load. Since even an 80GB H100 is an order of magnitude too small, distributed computing is a requirement rather than a choice.

In this post, we peel back the curtain on the distributed inference engine. We move beyond optimizing single chips to orchestrating clusters. We do this to break the HBM capacity ceiling and transform isolated cards into a single coherent brain.

## The Brutal Math of Model Weights

The primary driver of the capacity cliff is the raw size of the parameter set. To understand the "why" of multi-GPU deployment, you have to look at the storage requirements:

- **Llama 3.1 405B (FP16):** At 16-bit precision, this model requires approximately 810 GB of VRAM just to load the weights.
- **The Hardware Limit:** A single NVIDIA H100 GPU offers only 80 GB of High Bandwidth Memory (HBM).
- **The Reality:** Even with the most efficient kernels, you are nearly 10x short of the memory required to host the model on one chip.

## The Hidden Pressure: The KV Cache

It isn't just about the weights. As we push toward massive context windows (128k to 1M tokens), a second memory consumer emerges: the Key-Value (KV) Cache.

- **Why it matters:** The KV cache stores intermediate representations to avoid re-calculating the entire prompt history for every new token.
- **The Scaling Problem:** For a model like Llama 3.1 405B at a 128k context, the KV cache alone can consume hundreds of gigabytes, potentially rivaling the size of the model weights themselves.

## The Decision: Why and How We Scale

When you hit this cliff, your architectural strategy must shift from "How do I make this faster?" to "How do I make this possible?".

- **Mandatory Scaling:** If the weights and the projected KV cache exceed the total VRAM of your node (typically 640 GB for an 8x H100 setup), you must scale out to multiple nodes.
- **The Quantization Escape:** One way to delay the cliff is quantization. By moving from FP16 to FP8, you can shrink that 810 GB model down to ~405 GB, potentially allowing it to fit back into a single 8-GPU node.

However, for high-fidelity research or extreme context lengths, even quantization won't save you. At this point, the "Network" becomes the new "Memory Bus," and we must begin the complex work of sharding the brain across the cluster.

## Tensor Parallelism: The Intra-Node Sprint

**The Why: Latency and Bandwidth Aggregation**

In Part 1 we established that a single chip is trapped by its internal memory bus. Tensor Parallelism (TP) breaks this limit by turning a cluster of GPUs into one virtual processor. You use TP when your primary goal is the lowest possible latency for a single user. By splitting the work you are aggregating the memory bandwidth of every chip in the node. This allows the system to move data to the compute cores faster than any single chip ever could.

**The How: Horizontal Sharding**

Instead of putting whole layers on different chips we chop the weight matrices themselves. Every GPU calculates a portion of the same layer at the same time. In a typical Attention block some GPUs handle specific heads while others handle specific rows of the MLP. By splitting the math across 8 GPUs you are essentially creating one giant GPU with 8x the memory bus width.

**The Catch: The All-Reduce Sync**

Because each chip only calculates a partial result they must stop and synchronize before moving to the next layer of the model. This is the All-Reduce operation. In a frontier model like Llama 3.1 405B this synchronization happens hundreds of times during a single forward pass. Every GPU must wait for its neighbors to finish before it can proceed.

**The Hardware Constraint: NVLink**

This synchronized dance only works if the wires between the chips are as fast as the memory bus inside the chips. This is why TP is almost exclusively used inside a single server node. You need the massive 900 GB/s bandwidth of NVLink to share data in real time without stalling the compute cores. If you try to run TP over standard networking the communication overhead will consume the entire cycle and your performance will collapse.

## Pipeline Parallelism: The Inter-Node Relay

**The Why: Scaling Beyond the Box**

Tensor Parallelism is limited to the speed of the local fabric inside a single server. But when a model is so massive that it cannot fit into the VRAM of a single 8-GPU node (like Llama 3.1 405B in full precision), you must cross the "Inter-Node" boundary. Pipeline Parallelism (PP) is the strategy we use to bridge multiple physical servers into one coherent system.

**The How: Vertical Sharding**

Unlike Tensor Parallelism which chops layers horizontally, Pipeline Parallelism splits the model vertically by layers. GPU 0 handles layers 1 to 20, GPU 1 handles layers 21 to 40, and so on. The data flows through the system like an assembly line. Each GPU finishes its assigned chunk of the "brain" and passes the activations to the next GPU in the chain.

**The Catch: The Pipeline Bubble**

The biggest challenge with PP is sequential dependency. Because GPU 1 cannot start working until it receives the output from GPU 0, it spends a significant amount of time sitting idle. This idle time is known as the "Pipeline Bubble." To minimize this, we use micro-batching (shredding the input into smaller pieces) to keep as many GPUs as possible working at the same time.

**The Hardware: Surviving Slower Wires**

The reason we use PP for inter-node scaling is that it requires much less communication than TP. In Tensor Parallelism, the GPUs must synchronize hundreds of times per layer. In Pipeline Parallelism, they only talk once at the end of their assigned block of layers. This lower communication frequency allows PP to function effectively over standard data center networking like InfiniBand or even high-speed Ethernet, where NVLink is not available.

## Sequence Parallelism: Conquering the Context Wall

**The Why: Sharding the Memory of the Sequence**

As context windows expand to 128k or 1M tokens, the KV cache becomes a massive weight that a single GPU cannot carry. Even if the model weights fit, the activation tensors and the cache for a massive sequence will trigger an out of memory error. Sequence Parallelism (SP) is the solution for the "Context Wall." We use it to distribute the memory load of a single long sequence across multiple GPUs.

**The How: Sharding the Dimension**

In Tensor Parallelism we shard the weights. In Sequence Parallelism we shard the input itself. We take the long string of tokens and split it across the cluster. Each GPU is responsible for the attention math of only a fraction of the total sequence. This allows us to process prompts that are physically larger than the VRAM of any one device.

**The Catch: The Ring of Data**

Because self-attention requires every token to look at every other token, the GPUs cannot work in total isolation. They must share their local KV pairs with their neighbors to complete the math. We often use a technique called Ring Attention. The data circulates through the cluster in a ring, allowing each GPU to compute attention on the full sequence without ever having to store the whole thing.

**The Strategy: Hiding the Latency**

The secret to effective Sequence Parallelism is overlapping. While the GPU is doing the heavy math for its local chunk of the sequence, it is simultaneously sending data to the next chip in the ring. By hiding the communication behind the computation, we can scale context length almost indefinitely as long as we have enough chips to hold the shards.

## Expert Parallelism: Sharding the Specialists

**The Why: Extreme Model Capacity**

Mixture of Experts (MoE) models allow us to scale parameter counts into the trillions without a proportional increase in the compute cost per token. Because only a small subset of "experts" is active for any given token, we can have a massive model where most of the brain is dormant at any one time. Expert Parallelism (EP) is the strategy we use to store these hundreds of specialists across a cluster.

**The How: Sharding the Experts (Not the Tensors)**

In Tensor Parallelism, you slice every weight matrix into pieces. In Expert Parallelism, you keep the experts whole but place them on different GPUs. If a model has 64 experts and you have 8 GPUs, each GPU holds 8 full experts. This allows each chip to act as a dedicated host for a specific subset of the model's knowledge.

**The Catch: The Routing and All-to-All Sync**

The complexity of EP lies in the Router. For every token, the model must decide which expert is the best fit. If the chosen expert lives on GPU 4 but the token is currently on GPU 1, the data must be physically moved. This creates an "All-to-All" communication pattern where every GPU is potentially swapping tokens with every other GPU in the cluster.

**The Strategy: Balancing the Load**

EP is highly efficient for memory because it avoids the massive synchronization overhead of TP for every single layer. However, it introduces the risk of "hot experts." If one specific expert is popular (e.g., the one that handles common grammar), the GPU hosting it will be slammed while the others sit idle. Modern inference engines use sophisticated load balancing to ensure that these specialists are distributed such that no single chip becomes a bottleneck.

## The Topology Hierarchy: Matching Software to Wires

**The Why: The Physical Speed Limit**

In Part 1 we learned that the bottleneck inside a single GPU is the memory bus. When you move to multiple GPUs you introduce a new hierarchy of bottlenecks. You must understand the speed of your wires because your software strategy is dictated by the physical path the data has to travel. If you choose the wrong parallelism for your hardware the communication overhead will destroy your performance.

**The How: The Speed Hierarchy**

Think of GPU communication in three distinct tiers of speed and latency.

1. **High Bandwidth Memory (HBM):** This is the memory inside the chip. It moves data at approximately 2 to 3 TB/s. This is your gold standard for speed.
2. **NVLink (Intra-Node):** This is the specialized fabric connecting GPUs inside a single server. Moving data extremely quickly with new B200 NVLink 5 getting up to 1.8 TB/s (Although most versions will be up to 900 GB/s). This is fast enough to support the constant synchronization required by Tensor Parallelism.
3. **InfiniBand or Ethernet (Inter-Node):** This is the network connecting different server boxes. Even high end InfiniBand tops out around 50 to 100 GB/s. This is significantly slower than NVLink and is why we use Pipeline Parallelism here to minimize the number of times we have to "talk" across the wire.

**The Catch: The Latency Penalty**

It is not just about bandwidth. It is about latency. Every time a GPU has to send data to a neighbor it incurs a time penalty. Crossing from one chip to another inside a node is fast. Crossing from one server rack to another is slow. If your parallelism strategy requires 200 synchronizations per second but your network can only handle 50 you will face massive stalls.

**The Strategy: Topology Awareness**

Modern inference design is topology aware. We keep the chatty strategies like Tensor Parallelism inside the NVLink domain. We push the less frequent communication like Pipeline Parallelism to the Inter-Node domain. By matching the "Why" of your sharding to the "How" of your wiring you ensure that the network remains a tool rather than a bottleneck.

## The Orchestration Layer: Turning Theory into Production

**The Why: Managing Complexity at Scale**

You do not want to manually write the MPI calls or handle the CUDA kernels to shard every tensor. As a developer your goal is to serve the model efficiently while staying focused on the application. The orchestration layer exists to hide the mess of parallelism behind a clean API. These frameworks automate the sharding strategies we discussed and optimize the execution based on your specific hardware topology.

**The How: vLLM and TensorRT-LLM**

Modern inference engines like vLLM and TensorRT-LLM are the "operating systems" of the distributed cluster.

- **vLLM:** This engine uses PagedAttention to manage the KV cache and Ray to orchestrate the distributed workers. You simply tell it the tensor parallel size and it handles the sharding across your GPUs automatically.
- **TensorRT-LLM:** This is the high performance option from NVIDIA. It builds specialized engines that are deeply optimized for the underlying hardware. It allows for aggressive layer fusion and custom kernels that minimize the overhead of moving data between the GPUs.

**The Catch: The Configuration Struggle**

The difficulty lies in finding the "Goldilocks" configuration. There is no one size fits all setting. You have to balance your tensor parallel (TP) size and pipeline parallel (PP) size based on your specific latency and throughput requirements. If you set TP too high you might hit a communication bottleneck. If you set it too low you might run out of VRAM for your context window.

**The Strategy: Profile Before You Deploy**

The best way to use these tools is to profile your specific workload. You should test different parallelism combinations to see where the throughput saturates. Most modern engines provide detailed logs showing exactly how much time is spent on compute versus communication. Use these metrics to ensure that your sharding strategy matches the physical wires in your rack.
