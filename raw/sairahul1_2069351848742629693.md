# Twenty core ideas behind every AI agent

**Author:** @sairahul1

## Thesis

You don't fall behind by missing the latest tool. You fall behind by not understanding the ideas the tools are built on. Learn these concepts once, and every new framework becomes familiar.

## 1–4 Building blocks

**1. Agent** — Loop: goal → think → act → observe → repeat. Use for unpredictable feedback; use prompt or script when steps are fixed.

**2. Execution loop** — Think → act → observe; recovery from wrong steps is the power. Variants: parallel tools, blocking vs non-blocking.

**3. Agent state** — Context window + external storage (files, DB). Model only sees context; files are the default for dev work.

**4. Common patterns** — Planner/executor; router/specialist; map-reduce. Clean handoffs matter.

## 5–8 Configuration

**5. Agent config files (CLAUDE.md / AGENTS.md)** — Project rules; short and specific (<~100 lines).

**6. Reusable workflow files** — Task guides loaded on demand; human-written beats AI-generated; cheaper model + workflows can beat stronger model alone.

**7. Prompt caching** — Stable prefix not reprocessed each turn; doesn't fix bad context.

**8. Context rot** — Noise in context degrades focus; lost-in-the-middle; every token must earn its place.

## 9–11 Capability

**9. MCP** — Standard tool exposure; deferred loading to avoid huge schemas in context.

**10. Live document retrieval** — Current docs/repo state vs stale training data.

**11. Persistent memory** — Survives sessions (e.g. MEMORY.md, searchable memory).

## 12–13 Orchestration

**12. Subagents** — Focused agents, limited tools, fresh context; main thread gets summaries only.

**13. Agent loops** — Repeat with fresh context; progress in files/Git; good for bounded repetitive work.

## 14–18 Guardrails

**14. Sandboxing** — Limit read/write/connect (often Docker).

**15. Permissions** — Safe defaults + deny list (.env, destructive commands).

**16. Hooks** — Pre-tool checks for dangerous commands.

**17. Prompt injection defense** — Treat configs and external input as untrusted.

**18. Pre-commit gates** — Block bad code; agents respond to strict automated feedback.

## 19–20 Observability

**19. Tracing** — Tree of tool calls, timing, reasoning.

**20. Metrics** — Latency, tokens, failures, plus outcome metrics (tests, PR, deploy).

## Layers summary

Building blocks → configuration → capability → orchestration → guardrails → observability.

## Where to start

1. Solid CLAUDE.md / AGENTS.md  
2. Sandboxing  
3. Pre-commit gate  
4. One focused subagent task  

The tools change; these patterns don't.
