## Algorithm Answers

1. **Why Actor-Critic vs pure Critic?** Pure value-based methods can't handle large or continuous action spaces well and only give you a greedy policy. Pure policy gradient handles them but has high variance updates. Actor-critic keeps a direct policy and uses the critic to cut that variance. Best of both.

2. **KL divergence / cross entropy / MLE relationship?** CE(p,q) = KL(p,q) + entropy(p). Minimizing any of them lands you in the same place. MLE is just KL between data distribution and model distribution.

3. **How should rewards be designed?** Mostly depends on the env you're building. Verifiable stuff (math, code) is much easier to reward than non-verifiable (writing) for obvious reasons. Main thing: the env shouldn't be prone to reward hacking. There shouldn't be an easy path to optimizing the wrong solution. Data and rewards have to be written carefully.

4. **Importance sampling / rejection sampling / Monte Carlo in RL?** Importance sampling is used in async RL to reuse off-policy samples generated for an earlier policy of the trainer. Rejection sampling is for data filtering — you can filter SFT data or drop too-easy/too-hard ones. Broadly it's all Monte Carlo, estimating expectations from samples instead of computing them exactly.

5. **PPO vs GRPO advantage computation?** PPO has a critic and a reward model. GRPO drops both. Its "critic" comes from doing multiple rollouts and taking the mean as baseline. The baseline reduces variance so not every rollout gets a positive push. Kind of a zero-sum thing within the group. Std normalization isn't necessary — see Dr.GRPO.

6. **RL training vs test-time scaling exploration?** RL is about embedding domain intelligence into the weights. Test-time scaling is, given that intelligence, finding the best path to a solution and exploring different ones with more thinking tokens. One shapes the model weights while the other spends more on inference budget.

7. **PPO clipping?** Clipping is there because you don't want abrupt policy turns — the ratio variance across examples gets high. The min comes from TRPO's trust-region idea. It keeps updates pessimistic so you only move when it's favourable. CISPO is different — it clips the importance weight instead of the whole objective, so gradients still flow through all tokens and you keep more signal.

8. **GRPO KL penalty?** KL is there because you already have a good base model with intelligence you don't want to move too far from. But you also want to learn new domains via RL. It gets dropped because that same constraint stops you from moving far enough to actually learn. Removing KL helps stability and training in RLVR.

9. **Loss All Reduced multiple times?** Not sure.

10. **DPO reward function?** DPO has an implicit reward baked into the dataset — examples marked good vs bad — so no separate reward model needed like PPO. Reward hacking still happens because there's a backdoor to the solution you're chasing. It's an active problem with no single fix. The env has to be stress-tested hard. You can catch it via signals like sudden reward jumps or rollout length blowups.

11. **Train-inference mismatch in MoE?** Comes from: different engine, different expert selection, different kernels, quantization, rounding (see the Thinking Machines blog). One fix is replaying expert selection so inference and trainer pick the same experts. Broadly you're trying to make the rollout policy and the trained policy numerically identical — otherwise your importance weights are wrong.

12. **Group size / LR / PPO epochs / generation length?** Group size is just more rollouts, more paths to learn from — 8/16 usually. LR is manageable, 1e-6 range. PPO epochs is usually 1 — more epochs make the data more off-policy and destabilize things. Generation length is purely task-dependent.

13. **GRPO variants compared?** Dr.GRPO: remove std. DAPO: clip-higher, dynamic sampling, token-level loss, overlong filtering. GSPO: sequence-level importance sampling instead of token-level, much better for MoE. CISPO: clip the weight not the objective. MaxRL: solves diversity collapse. Not sure about SAPO/DPPO (haven't read the papers).

14. **TRPO / DPPO / AReaL trust-region constraints?** TRPO is constrained optimization, staying within a KL boundary. PPO does the same thing approximately with the clip. AReaL enforces it in the async setting by bounding staleness/off-policyness instead of a hard KL.

15. **Can RL expand capability frontier?** Yeah, the papers suggest RL mostly sharpens abilities already in the base model rather than forming new ones. Cool direction to push on though — with enough exploration and diversity, not fully sure it can't.

16. **ProRL?** Haven't read the paper.

17. **OPD improvements?** OPD combines RL's on-policy exploration with SFT's dense signal. RL's main weakness is sparse reward — OPD fixes that by having a teacher score the student's own (on-policy) rollouts (important part). Used for cheaply transferring capabilities. Been on the trend lately.

18. **When does reasoning emerge?** Not fully sure, but CoT plays a big part. Also pre-training for baseline intelligence and post-training for amplifying the intelligence using reasoning and test-time compute.

19. **DeepSeek R1→V3.2→V4 RL improvements?** Don't remember the V4 stuff.

## Infrastructure Answers

1. **Model copies in memory during GRPO?** Depends mostly on the algo and KL. For GRPO with KL you need: training policy, reference policy for KL, and the inference model. For memory optimization: start by dropping KL (removes extra ref model). NVFP4 quantization saves memory. Properly distributed training helps a lot — FSDP, EP, TP split model and optimizer states across GPUs. You can also quantize the inference copy and shard optimizer states so each rank holds only a slice.

2. **KV cache transfer?** KV cache transfer matters for disaggregated prefill or decode. Optimized via NVLink or layer-wise overlapped transfer of KV/weights. Multi-GPU depends on whether same node or different node. Generally TP helps a lot and you want to keep heavy KV movement within a node.

3. **INT8 vs FP8?** FP8 has more exponent bits — better for training. INT8 has more precision in a fixed range — better for inference weights. So FP8 for training, INT8 for inference serving.

4. **Long-tail rollouts?** Addressed with methods like PipelineRL or continuous batching or early truncation. Mostly seen in async setups.

5. **Continuous batching in RL?** Continuous batching mixes decoding for rollouts at different stages of next-token prediction. vLLM uses paged-attention, SGLang uses RadixAttention (prefix sharing). The RL thing here is aligning complete trajectories and logprobs when sequences finish at different times.

6. **Utilization measurement?** Measured via throughput (tok/s), KV cache util, and GPU util. For KV cache you watch occupancy. Also watch for memory bound. Rollouts getting kicked out. Low GPU util usually means you're waiting on something (rollout sync or CPU) rather than compute-bound. Both engines expose scheduler/cache metrics.

7. **Backpropagation in multi-node RL?** Same as normal backprop with a few tweaks: basically FSDP, EP and TP. For long context it's CP (context parallel). Ideally no one uses PP — it's complex and not really necessary at most scales.

8. **Async RL frameworks?** Prime-RL, VeRL, and TRL. The bottleneck comes down to three things: staleness of rollouts and how you manage it (importance sampling + staleness bounds), trainer-inference mismatch, and token-in token-out. For mismatch: make rollout engine and trainer numerically consistent, replaying expert/router decisions and computing logprobs on the trainer side rather than trusting the inference engine.

9. **AReaL KV cache preservation?** Some methods preserve the old KV cache and some don't. No real perf difference between the two.

10. **Expert Parallelism in MoE?** EP plays a big role for MoE models — a bunch of experts are placed on a node or set of GPUs so each holds a subset. Raises throughput by parallelizing the expert FFNs, but adds all-to-all communication to route tokens to the right expert. The actual gain: all-to-all overlaps with compute and how balanced the expert load is.

11. **Compute-communication overlap?** Parallelizing communication along with computation is the key. FSDP shards grads/params/optim, gathers them on demand — much simpler. Megatron uses DP, PP and TP (3D parallelism) with more explicit comms.

12. **Deterministic execution / batch invariance?** Needs reduction order, same kernels, and batch invariant ops. Refer the Thinky blog for more.

13. **AReaL vs slime?** AReaL goes fully async with staleness control, decoupling rollout generation from training so the trainer never waits on the slowest rollout. Has ways of controlling staleness using max off-policy steps, importance sampling, or removing samples after threshold policies. Not sure about slime — need to read more.

14. **Staleness in async RL?** Staleness is the gap between the policy that generated rollouts vs the policy you're training. Usually 1-4 steps behind. More staleness leads to instability in importance sampling since ratios blow up.

15. **Slime data flow?** Not sure.

16. **Which framework?** None of them — use @PrimeIntellect.
