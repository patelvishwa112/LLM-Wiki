# What is KV Cache in LLMs?

LLMs generate text one token at a time. Each new token requires looking at all previous tokens. Inside the attention layer, every token is converted into Query (what the token is looking for), Key (what information the token holds), and Value (the actual information).

**The problem: repeated computation.** At step 1, K/V computed for 2 tokens. Step 2: 3 tokens (2 recomputed). Step 3: 4 tokens (3 recomputed). Step N: N+1 tokens computed, N recomputed. For 100 tokens: 5,049 computations total. The model keeps redoing work for tokens it's already seen.

**KV Cache: compute once, store, reuse.** Step 1: compute K/V for all tokens, save to cache. Step 2+: compute K/V only for the new token, retrieve the rest from cache. For 100 tokens: 101 computations instead of 5,049 (~50x fewer). The longer the sequence, the bigger the savings.

**Why only K and V?** Query is only needed for the current token being generated. Once prediction is done, that Query is discarded. But Keys and Values of past tokens are needed at every future step.

**The trade-off: speed vs memory.** KV Cache uses extra memory to store K/V for all generated tokens. For very long sequences (thousands of tokens), the cache can consume significant memory. But the speed improvement is worth it for most use cases.
