# What if a Transformer could think twice?

**Author:** Kaizhao Liang (@KyleLiang5)
**Date:** 2026-07-18
**Source:** https://x.com/kyleliang5/status/2078543088419840292
**Project:** https://kyleliang919.github.io/looped_nanochat/

What if a Transformer could think twice?

Most language models scale in familiar ways:
- more parameters
- more data
- more test-time tokens

Looped Nanochat explores a different idea:
What if the model could revisit the same sequence, using what it learned in the first pass to guide the second?

That is the core idea behind Looped Nanochat, a two-pass Transformer built on @karpathy's nanochat.

## How it works

The model runs the same Transformer trunk twice.
The weights are shared across both passes, so this is not simply two separate models stacked together.

During the first pass, the model processes the sequence normally.
At several deeper layers, it saves the query and key features that determine attention.

The second pass then starts again from the token embeddings.
But this time, each layer receives an attention prior from the first pass.

In simple terms:
attention logits = current scores + gate × first-pass prior

The model learns how strongly to use that prior at each layer.

## Why this is interesting

A normal Transformer only moves information forward through depth.
An early layer cannot directly use a representation discovered much later in the network.

Looped Nanochat creates a feedback path.
A deeper representation from the first pass can guide an earlier layer in the second pass.

So the model can:
- build an initial understanding
- use that understanding to decide where to look again

You can think of the first pass as forming a rough plan, and the second pass as refining the computation around that plan.

## The routing pattern

In the 24-layer model, the second pass uses three routing sources:
- layers 0–7 use the prior from first-pass layer 12
- layers 8–15 use the prior from first-pass layer 17
- layers 16–23 use the prior from first-pass layer 23

This creates a progressive routing pattern.
Earlier layers in the second pass get guidance from relatively deep first-pass representations, while later layers use even deeper ones.

The learned routing gates are strongest in the early second-pass layers and gradually become weaker with depth.
That suggests the model relies most on first-pass guidance when rebuilding its early representations.

## Almost no extra parameters

The two passes share the same Transformer weights.
The only new parameters are:
- one small routing gate per layer
- two pass embeddings

So the model has almost the same parameter count as the baseline (~1.38B).
The difference is the computation graph.
Same weights, richer computation.

## Results

Trained baseline nanochat and looped version at the same scale:
- 24 layers
- about 1.38B parameters
- about 5.84B pretraining tokens
- the same data
- the same optimizer schedule
- the same post-training setup

The looped model improved or tied on every reported benchmark.

Some highlights:
- Base CORE: +9.0% (0.2552 → 0.2783)
- ChatCORE: +11.9% (0.2253 → 0.2521)
- ARC-Easy: 62.04% → 68.01%
- ARC-Challenge: 50.00% → 52.22%
- MMLU: 37.06% → 37.96%
- GSM8K: 2.88% → 4.17% (~45% relative)
- HumanEval: unchanged (10.98%)

The biggest relative gain was on GSM8K. Absolute GSM8K is still low at this size — do not overstate.

## The compute cost

Not free:
- ~2.4× baseline training time
- ~2× single-pass decoding latency

Parameter-efficient, not compute-efficient.
Next test: iso-FLOP vs larger/longer single-pass model.

## A useful negative result

Running more than two passes at inference (no retraining) hurts.
Trained for exactly two passes; K=2 best; extra untrained passes degrade validation bpb.
Recurrence is not automatically a test-time scaling method.
Would need variable-pass training, curriculum over pass count, or explicit iterative-refinement objective.

## Open questions

- Does this scale to larger models?
- Does it help more on reasoning-heavy tasks?
- Can the routing become input-dependent?
- Can the model learn to use a variable number of passes?
- Can the extra compute cost be reduced?
- Is looping better than simply making the model deeper?

## Project page technical notes (Firecrawl)

Injection without materializing N×N: concatenate q/k along head dim to width 2D so one attention equals softmax(q2k2^T + g·q1k1^T)·v2. Gate g = softplus(gate_logit). Early layers g ≈ 0.75–0.87; decays to ~0.2 deep.
