# Prompter vs loop designer (10-step Claude Code roadmap)

**Author:** @de1lymoon  
**Posted:** June 24, 2026

## Post text

Most people talk to Claude Code. They type a request, watch it work for a few minutes, read the result, and type the next request. They are prompters.

The agent is a very capable tool that sits idle until poked.

A **loop designer** builds something different: a system that prompts itself. It runs on a timer, checks its own work against a goal, spawns helpers when it needs them, and writes down what it learned so the next run starts smarter.

The designer is not in the chair for most of it. They built the thing that sits in the chair.

The gap between the two is not talent and it is not a better prompt. It is ten moves, and none of them is exotic.

This is the roadmap: three steps to see the loop clearly, four to build it, three to make it compound instead of bleed. Everything here uses Claude Code parts you already have.

### Tier 1 · See the loop

**01. A loop is a prompt on a timer**  
Strip the mystique — a loop is one idea: instead of *you* sending the next prompt, the *system* sends it. It runs the agent, looks at the result, decides whether the job is done, and if not, runs it again.

A while loop with a model inside.

**Prompt → Agent runs → Check vs Goal → Done (repeat)**

This reframing is the whole shift from prompter to designer. A prompter optimizes the single message. A designer optimizes the *cycle*.

**02. The harness comes first.**  
A loop is only as good as the environment it runs in (the model, tools, permissions, and starting context). Get one manual run reliable first — with a good `CLAUDE.md`, clear verification target, and right tools. Weak harness + loop = garbage produced faster.

**03. Self-improving is the system, not the model.**  
The model’s weights don’t change. What improves is the *system* around it: accumulated memory, sharpened skills, and better grader. You’re not waiting for the model to get smarter — you’re building an environment that gets smarter run over run.

### Tier 2 · Build the loop

**04. Set a goal and an independent grader.**  
The loop needs a stop condition that isn’t “the agent feels finished.” Use an independent checker (not the one that did the work).

**05. Split the maker from the checker.**  
A separate agent with its own fresh context judges only the artifact against the standard. This avoids the model preferring its own earlier reasoning.

**06. Put it on a timer, then in the cloud.**  
Use `/loop` to rerun on an interval. Then move it to cloud routines so it runs on Anthropic’s infrastructure even when your laptop is closed.

**07. Compose the hard ones with workflows.**  
For complex jobs, have the system create orchestration plans (fan-out + synthesize, adversarial verification, or loop-until-stop).

### Tier 3 · Make it compound

**08. Give the loop a memory.**  
Use a state file that every run reads at the start and updates at the end. This prevents restarting from zero.

**09. Distill lessons into skills.**  
General lessons that apply beyond the current project should graduate from the state file into reusable skills/procedures that future projects inherit.

**10. Close the loop, and make it fail safe.**  
Add guardrails/hooks that the model cannot talk its way past. Route work by cost (heavy model for orchestration, cheaper ones for volume). Make sure unattended loops cannot do anything irreversible.

### Common mistakes that prevent compounding

- Looping a thin harness
- Letting the maker grade itself
- No independent stop condition
- No memory / state file
- Lessons that stay trapped in the project state file
- Unattended loops with broad permissions
- Using top-tier model for every single iteration

### Closing

Stop optimizing single prompts. Start designing loops.

A prompter operates a powerful tool by hand.  
A loop designer builds a system that operates *itself* and only calls the human for goals, standards, final merges, or irreversible decisions.

The move is a sequence: see the agent as a loop → build the harness → add goal + independent grader → timer/cloud → memory + distillation.

Pick the one step you’re missing today (probably an independent grader, state file, or safety hook) and add it.
