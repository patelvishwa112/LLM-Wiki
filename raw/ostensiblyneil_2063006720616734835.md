# Improving LM Studio's MLX Engine for Agentic Workflows

By Neil Mehta (@ostensiblyneil), LM Studio

MLX Engine v1.8.5 dramatically improves performance for repeated, long-context agentic workflows by checkpointing the KV cache. Adds continuous batching for VLM requests. MIT-licensed, open source.

## What is mlx-engine?

MLX Engine is an MIT-licensed inference engine optimized for Apple silicon, created and maintained by LM Studio. Uses Apple's MLX library, builds on mlx-lm and mlx-vlm. It's LM Studio's backend for all MLX inferencing.

## The Cache-Reuse Problem

Popular open-source models (Qwen 3.5/3.6, Gemma 4) use architectures that reduce KV cache size at large context lengths but make the cache not arbitrarily rewindable:

- **Qwen 3.5**: Hybrid architecture
- **Gemma 4**: Sliding window — interleaves "local" attention (512-token window) and "global" attention layers

Gemma 4 inference flow:
1. **Prompt prefill** — compute KV cache for system prompt + user message
2. **Decode** — build KV cache while computing assistant reasoning + response
3. **Rewind** — trim KV cache to step 1, append assistant message without prior reasoning

The problem: avoiding re-computation when rewinding the KV cache for follow-up requests.

## How mlx-engine v1.8.5 Solves It

### Disk-Backed KV Cache

**Saving:** At every 256-token boundary, stream a copy of local attention layers' KV cache to a disk-writer backend. A background process persists blocks to disk. Since Apple silicon has unified memory, the engine commits local attention KV cache to disk and evicts it from memory — memory footprint scales with active sequences, not all previously seen sequences.

**Restoring:** Calculate a key for each 256-token block, determine which global and local blocks need retrieval, load as much as possible from disk. For prompt sections never computed or evicted, schedule prompt prefill. The disk cache is an LRU store — short prompts reusing the same system prompt keep their cache, stale conversation caches get evicted.

### Disk Cache Design

- One scratch file (not a folder of files)
- Each entry is a serialized safetensors blob
- In-memory table: "entry X at byte offset Y, Z bytes long"
- Evicted byte ranges returned to free list, reused
- Temporary: stored in /tmp, metadata is model-lifetime only
- On model unload: cache clears index, closes scratch file. OS releases storage if process exits.

### Continuous Batching for VLMs

Concurrent request processing for vision models. Along with KV cache improvements, mlx-engine can now handle serious agentic workloads.

## Benchmarks (M3 Max MacBook Pro, 36GB RAM, Qwen3.6-27B-MLX-4bit)

| Benchmark | Before | After | Improvement |
|-----------|--------|-------|-------------|
| Parallel chat (4-way) | 37.60s | 16.78s | **2.2x faster** |
| Parallel long prompts RAM | +6.47 GB | +1.18 GB | **82% less extra RAM** |
| Repeated image prompt (2nd) | 23.79s | 6.88s | **3.5x faster** |
