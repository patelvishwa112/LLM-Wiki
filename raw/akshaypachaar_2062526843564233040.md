# Hermes Kanban: Mission control for your Agents

A complete walkthrough of building a 4-agent software team (PM, backend, frontend, tester) in about 20 minutes using Hermes Kanban. Agents coordinate via a shared task board, each handing off structured context to the next — no cold starts, no context loss.

**The demo:** A working Google Docs clone, AI features included, built entirely by the agent team.

## Why one agent isn't enough

Adding a password reset feature splits work across three domains: backend (REST API), frontend (UI), tester (E2E verification). Running all three through a single agent fills the context window — the agent loses track of what it did three steps ago. Splitting across agents creates the shared-context problem: how does the frontend know what the backend built? **Hermes Kanban solves shared context.**

## The mental model

A task board where agents sit behind columns instead of humans. Every task survives crashes and reboots. When an agent finishes, it writes a summary: files changed, what it built, what the next agent needs. The next agent reads that summary before starting. **That summary is the whole insight — it turns agents into a team instead of strangers.**

## The vocabulary (6 columns)

- **Triage:** Rough ideas before full spec
- **Todo:** Created, waiting on dependency
- **Ready:** Dependencies met, waiting for agent
- **In Progress:** Agent actively running
- **Blocked:** Hit a wall, flagged for human
- **Done:** Finished with full run history preserved

## The orchestrator

A project manager agent that reads high-level goals, checks which agent profiles exist, breaks goals into linked dependency chains, and steps back. Never writes code — coordinates.

Rule: check profiles exist before creating tasks. Tasks with non-existent assignees are silently skipped.

## Three patterns

**1. Pipeline:** Each task depends on previous one. Backend completes → frontend moves to ready with backend's work attached. Chain via `--parent`.

**2. Human in the loop:** Agent calls `kanban_block(reason="...")` when it finds a real problem. You unblock via the dashboard or Telegram. The retry reads the block reason and knows exactly what to fix.

**3. Triage specifier:** Drop a rough idea with `--triage`. `kanban specify` rewrites a three-word title into a full task with goal, approach, and acceptance criteria.

## Setup (4 profiles, ~20 minutes)

```bash
# Create profiles
hermes profile create project-manager --clone
hermes profile create backend-developer --clone
hermes profile create frontend-developer --clone
hermes profile create tester --clone

# Add SOUL.md to each profile with role-specific instructions

# Install skills
hermes skills tap add InsForge/insforge-skills
hermes -p backend-developer skills install InsForge/insforge-skills/insforge --force

# Initialize and start
hermes kanban init
hermes -p project-manager gateway setup
hermes -p project-manager gateway start
```

## Dashboard

`hermes dashboard` at http://127.0.0.1:9119 — live WebSocket updates, lanes by profile, nudge dispatcher, click-to-inspect full run history.

## Watch out for

- **Database overload:** Cap concurrency with `--max 4` for 10+ agents
- **Scratch workspace deletion:** Use worktree for coding tasks
- **Local model saturation:** `--max 2` for local models (Ollama/GPU)
