if you haven't watched this video yet, go watch it first.
this article is basically my notes and takeaways from it!

soo using cursor to autocomplete a function isn't agentic engineering.
asking claude to build a landing page isn't agentic engineering.
even spending your entire day prompting an ai model isn't agentic engineering.
btw kun is a former principal engineer at microsoft, meta and atlassian.  he ships 40 to 50 production pull requests almost every day with ai agents.
that's the part that caught my attention.
because if someone can consistently ship at that speed without sacrificing quality, there's probably something worth learning.
after watching both of his workflow videos.
one thing is sure that he treats his agents like an engineering team.
and i think that's exactly how software development is going to look over the next few years.

so basically this is all you need  : 

1. how to run multiple ai engineers in parallel
manage multiple claude code sessions with different effort levels, use git worktrees to avoid conflicts, and know when to bring in devin for long running tasks.
2. how to build a workflow that scales
learn when to use /compact vs /clear, prevent context drift, and keep your agents productive across long coding sessions.
3. how to make ai write better code
use the smell command, clean code principles, design patterns, and know when to use mcp servers, bash tools, or skills to connect external services.


you're not the programmer anymore!!!!
this was probably the biggest mindset shift for me. most people open claude or cursor and think, including me that :
"how do i get ai to write this code?"
he thinks completely differently.
"how do i manage a team of agents so they build this for me?"
that sounds like a small difference but it isn't.
>an engineering manager creates systems. they make sure everyone understands the task, review the work, keep quality high.
that's exactly what you're doing with ai.
>if your agent keeps making the same mistakes, that's not because the model is bad. it's because your onboarding process is bad fr.
build the system before you build the product
one thing i really liked about kun's workflow is that he spends a surprising amount of time building his own engineering environment.
most people obsess over which model is better.
he barely talks about that. instead, he optimizes the environment around the models. take notes guys!!
tmux manages multiple sessions.
fyi : tmux is a terminal multiplexer. It lets you switch easily between several programs in one terminal
link here : https://github.com/tmux/tmux/wiki
neovim keeps his hands on the keyboard. 
link here : https://neovim.io/
i didnt know this super cool. go get it.
memory files teach agents how he likes to work.
skills teach agents how to perform specific tasks.


your agents are new employees!!
every new claude session is basically a new engineer.
they know how to code, your company, your project, or your coding style.
instead of explaining everything again, build a memory layer.
every agent should start with:
>global memory files
>project memory files
>skills
>coding conventions
>testing instructions
memory is more valuable than prompting.
a good prompt solves one problem.
good memory solves every future problem.
you don't need to build everything yourself either.
context(dot)dev for long term project context
supermemory for remembering information across sessions
markdown memory files inside your repo for project knowledge
and one last thing.
don't rush into coding.
planning isn't wasted time.
the better your plan, the less you'll have to fix later.
and planning isn't wasted time!!
spend time discussing the feature first.
>what are the options?
>what should the ui look like?
>what happens in edge cases?
once the requirements are clear, implementation becomes surprisingly boring.
and honestly, that's exactly what you want.

run multiple agents in parallel.

one thing i'd also recommend adding to your workflow is @devinai
instead of thinking of it as another ai chat window, think of it as an autonomous software engineer that can own longer running tasks.
>while you're planning a feature, talking to users, reviewing code, or even away from your computer, devin can keep working in the background. 
>it can read your codebase, implement features, investigate bugs, open pull requests
>and iterate on feedback without needing constant supervision.
this changes the way you work.
instead of asking one model to do everything, you start assigning different responsibilities.
multiple agents are making progress at the same time while you stay focused on higher leverage work.
that's what agentic engineering is really about.

it's like building a team of ai engineers, where each agent has a clear responsibility and can work independently.
