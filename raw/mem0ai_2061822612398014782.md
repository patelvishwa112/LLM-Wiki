## Post

> Agent harnesses are where AI software actually runs. Cursor, Devin, Claude Code, Codex: these environments handle context, orchestrate tools, coordinate agents, and increasingly, manage memory. The harness, not the model, is increasingly the product that ships software.
>
> Memory is where harness design gets hard. Where does it sit? What persists when a session ends? It is a largely unsolved problem, and every major harness is solving it differently.

## What Agent Memory Actually Is

Three different things get called memory:
- Working memory: what lives in the context window during a session. Resets on session end; compaction problem belongs here.
- External memory: anything persisted outside the weights: vector stores, knowledge graphs, files. Survives sessions; weights don't change. Where essentially all production memory lives in 2026.
- Parametric memory: knowledge encoded into weights via gradient descent, shaped by training loop the harness feeds. Zero production deployments in 2026.

(Cognitive-science split: semantic / episodic / procedural describes what kind of information is stored; the three tiers describe where it lives.)

Paper "Contextual Agentic Memory is a Memo, Not True Memory" (arXiv:2604.27707) formalizes the ceiling: retrieval needs Ω(k²) stored examples to match what parametric memory does with O(d) weight updates.

## What Major Harnesses Shipped

### 1. Anthropic: Claude Code
Two tracks. CLAUDE.md = human-authored config read at session start. Auto-memory = Claude-written notes from background extraction agent, stored under ~/.claude/projects/<repo>/memory/ around MEMORY.md index capped at 200 lines / 25KB. Four categories: user, feedback, project, reference.

Retrieval: each turn makes separate call to smaller model with manifest of filenames + descriptions; model picks which files load. No embeddings, max five files per turn, silent truncation past cap.

Shortcoming: Selection by filename, not semantic search. Local, repo-scoped markdown with no semantic index.

### 2. Anthropic: Managed Agents
Session = append-only event log, never mutated (rollback and audit are architectural). Memory stores mount at /mnt/memory/ as filesystem (up to 8 per workspace, ~100KB each). Every write is immutable version; multiple agents can share one store concurrently with auditable history.

Shortcoming: Built for multi-agent coordination at workspace scale, not long-term personal memory. 100KB-per-store ceiling, workspace scoping.

### 3. OpenAI Codex
One directory of markdown: ~/.codex/memories/ (no SQLite, no embeddings). memory_summary.md read first, MEMORY.md grepped on demand, plus raw_memories.md, skills/, rollout_summaries/. Off by default behind features.memories flag.

Two-phase write: Phase 1 (per-rollout): after 6h idle, extracts against strict schema, redacts secrets, writes to local state DB. Phase 2 (global): under lock, consolidation sub-agent merges/patches/drops, writes diff. Bounded (256 rollouts), age-pruned (30 days), rate-limit aware.

Shortcoming: 5,000-token summary truncates silently; grep is substring-only; 6h idle gate means back-to-back sessions may never consolidate; local-only; unavailable in EEA/UK/Switzerland at launch.

### 4. GitHub Copilot
Distinctive move: just-in-time citation verification. Memory items are structured objects (subject, factual content, file-and-line citation, reasoning). Before use, agent validates citations against current branch, rewriting memories the code now contradicts. Auto-expire after 28 days.

Only deployed staleness mechanism with published outcome data: A/B tested (p<0.00001), PR merge rate 83%→90%, code-review precision +3%, recall +4%.

Shortcoming: Citation schema can't cleanly hold ungroundable or preference-based facts ("prefers minimal abstraction"). Strictly repo-scoped.

### 5. OpenClaw
Markdown in ~/.openclaw/workspace (curated MEMORY.md + dated daily logs) backed by per-agent SQLite index with embeddings and hybrid retrieval (70% vector, 30% BM25). Has semantic search natively.

Problem: when window fills, fires "silent internal turn" asking model to write important content to disk. What gets written is whatever the model decides in that one turn — long-term memory is selective and inconsistent.

Mem0 plugin fixes this: Auto-Recall injects relevant memories before each turn, Auto-Capture persists each exchange after. 247K-star adoption driven substantially by this gap.

### 6. NousResearch: Hermes Agent
135K stars, 200+ models. Three built-in layers plus 8 pluggable providers:
- Layer 1 (working memory): MEMORY.md (2,200 chars) and USER.md (1,375 chars), ~1,300 tokens combined. Consolidation at 80% capacity.
- Layer 2 (skills): procedural docs written after 5+ tool-call tasks, curated on schedule.
- Layer 3 (session search): SQLite FTS5 over all sessions, summarized on demand.

Shortcoming: Tiny cap (~800 tokens durable memory), FTS5 is keyword-only ("429 errors" won't match "rate limiting"), local. Mem0 provider removes cap, adds semantic retrieval, server-side extraction, user-scoped writes, circuit breaker.

### 7. AWS Bedrock AgentCore
Hosted agent platform. Memory runs three async extraction strategies (semantic facts, preferences, narrative summary), ~20-40s extract, ~200ms retrieve. Changed facts marked INVALID rather than deleted (preserving lineage). Published: LoCoMo 70.58, PrefEval 79, PolyBench-QA 83.02.

Shortcoming: AWS-specific lock-in, LoCoMo well below leading memory systems.

### 8. Windsurf
Memory generated and managed by Cascade engine, no developer workflow. Local, workspace-scoped at ~/.codeium/windsurf/memories/ capturing codebase patterns and conventions.

Shortcoming: Cascade's call, not developer's. Workspace-scoped, local only.

### 9. Cognition Devin
Two-part split. Knowledge = human-curated trigger-content facts (no auto-capture). DeepWiki = reference docs (30 pages, 100 notes, 10K chars each). Devin suggests Knowledge after sessions; human approves before storage.

Shortcoming: Approval gate = quality but friction. Teams that don't review accumulate nothing. Doesn't transfer to other tools.

## Memory Benchmarks Are the Weak Link

Benchmarks mostly test recall, are near-saturated, high scores don't predict better decisions.
- LoCoMo: worst of common ones. 10 conversations, many questions don't require memory (grep baseline ~74%), adversarial questions share surface similarity.
- LongMemEval: still okay. 500 curated questions across 5 abilities, scaling toward 1.5M tokens. Still recall-centric.
- MemoryArena (arXiv:2602.16313): tests memory that must guide action. Systems that near-saturate LoCoMo/LongMemEval fail.
- BEAM (ICLR 2026): only benchmark built for 10M+ token production scale.

"The honest conclusion: the field needs a new memory benchmark, and leaderboard scores should be read skeptically."

## What Research Shows Is Still Broken

- Stability-plasticity dilemma relocated. New and old memories compete for retrieval slots (forward transfer -9.5%).
- Selective forgetting unsolved. MemoryAgentBench names four competencies; systems handle retrieval but not unlearning stale facts while keeping surrounding structure.
- Memory is an attack surface. 57-71% cross-user contamination under normal usage; poisoning attacks succeed at 6-38%.

## The Pattern in the Shortcomings

Same gaps recur: storage bounded and local (Claude Code 25KB, Hermes 2,200 chars, Codex 5K-token load). Retrieval mostly keyword (Claude Code by filename, Codex by grep, Hermes by FTS5). Memory harness-scoped. Staleness handling barely exists (Copilot aside). Isolation is an afterthought.

## Where Things Stand

Memory is now infrastructure. Every major harness shipped it. Harness-native implementations made real progress but break at the same boundary: bounded local storage, keyword retrieval, harness scoping, weak staleness, isolation gaps. Benchmarks are weak. Mem0 fills the gap: portable, semantically searchable, cross-agent, scaled to production token volumes.
