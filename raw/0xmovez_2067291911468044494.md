# The Self-Improving Loop: a 300-agent swarm on Kimi K2.6, verified by Opus 4.8

A free open-source model is running 300 parallel agents across 4,000 coordinated steps from a single prompt, and it scores higher on real research tasks than models you pay 5x more for.

Most people have never opened it. They open Kimi, type a question, get an answer, close the tab. That's the chatbox. It works. It's also about 10% of what the product does.

Here's the part most people skip: The swarm doesn't just run fast. Run it right and it leaves something behind every time — a reusable skill, a sharper spec, a constraint that stops the next run from repeating today's mistake. The swarm that ran your task yesterday should be smarter than the one running it today.

That's the loop. Kimi does the work and the learning. Opus 4.8 sits at one gate — the verify gate — and its only job is to stop garbage from getting saved as a skill. The engine learns. The closer keeps it honest.

Some people pick one model and marry it. Some chase the top benchmark line. Others wire up LangGraph and spend a weekend debugging a DAG. The result is usually the same: a workflow that does the exact same thing on run #50 as it did on run #1.

This is not that. This is the complete playbook for a swarm that compounds. 10 steps. Every prompt is copy-paste. Every number is verified.

**Part 1 — Build the loop once. Run it forever.**

**01. Write a spec, not a prompt**  
Treat the swarm like a contractor, not a genie. A detailed spec defines what to collect, valid sources, exact output format, and conflict resolution. The spec becomes the seed of your reusable skill.

**02. Read the decomposition plan before you spend a cent**  
Kimi shows the execution plan (agent count, dependencies, step budget) before running. Review it — this prevents wasting money on poorly decomposed swarms. (Note: 4,000 steps is the *total* coordinated budget across ~300 agents.)

**03. Let it be wasteful — that's the point**  
Run up to 300 parallel sub-agents in waves. Each gets its own bounded context window, avoiding the quality collapse that happens when a single agent tries to do everything in one long context.

**04. Demand real files, not a chat answer**  
Specify exact deliverables in the spec (e.g. "40-page PDF + CSV with 20k rows + 14 export-ready PNG charts"). This dramatically improves output quality.

**05. Point the honest model at the output and ask what's wrong**  
Use Opus 4.8 purely as a verifier/refuter to catch flaws, under-cited claims, and contradictions before anything gets saved.

**06. Save the whole workflow as a Skill**  
After a successful run, capture the entire workflow (input format, agent steps, output format) as a reusable Skill. First run: ~20 minutes. Subsequent runs: ~30 seconds.

**07. Feed your own documents in as swarm knowledge**  
Upload your best past work so the swarm internalizes your style, structure, and domain knowledge.

**08. Turn the verify feedback into a permanent rule**  
Bake lessons from Opus's critiques into a project-level constraints file that every future run reads automatically.

**09. Replay the skill on new inputs — watch the cost collapse**  
Later runs inherit all previous improvements, skills, and constraints. The gap between run #1 and run #50 is the entire value of building the loop.

**10. Promote the loop to a background agent**  
Once stable, turn it into a proactive agent that triggers on schedules, new files, or competitor updates and only surfaces deviations/deliverables.

**Conclusion:**  
The real shift isn't which single model is smartest. It's how many you can run in parallel, how well you verify their work, and whether your system is meaningfully smarter today than it was yesterday.

Most will keep using Kimi as a chatbox. A few will build the loop this week.

Build it. Verify it. Distill it. Then watch it get cheaper and sharper every single time you run it.
