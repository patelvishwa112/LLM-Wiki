Let's learn about how does a GPU execute code from first principles.

The market is filled with engineers who can write code but how many know what happens inside the GPU once you run the code. Today in this blog, you'll learn it at once from first principles.

## GPU vs CPU

A CPU is like a mastermind polymath that can do anything and is almost good for a lot of operations. But the only thing is that it lacks in capacity of doing massive calculations together in less time.

A GPU is like a specialist that is built to do massively parallel calculations that are independent of each other. It can do a huge amount of calculations in less time (very less than a CPU).

A CPU is designed for low-latency execution and complex decision making. A GPU sacrifices that flexibility in exchange for massive parallel execution.

## What kind of calculations can a GPU do fast?

A GPU can do a lot of small independent operations at the same time by parallelly executing them. For example, consider a vector addition problem:

```
x[i] + y[i] = z[i]
```

If we do it by CPU, it will add the numbers one after another using its cores and threads. But a GPU can launch thousands of these lightweight threads together and parallelize this and get this done in 50x or 100x faster than the CPU.

Another case: `A[i] + B[i-1] = C[i]`. This case is dependent on fetching the last values of i-1 for B in order to calculate C[i] — which cannot be done efficiently by GPUs and using GPUs is an overkill.

> "GPUs work best when calculations are independent from each other. Dependent operations are harder to parallelize efficiently."

## Components of a GPU

To understand how a GPU executes code, we need to understand how work is organized inside the GPU. We'll build this from the smallest unit all the way to the full GPU.

### Thread — the smallest worker

A thread is the smallest worker that executes a calculation. In our vector addition example, one thread will execute `x[0] + y[0] = z[0]` and then another thread will execute `x[1] + y[1] = z[1]` and so on. A GPU can launch thousands of threads like this one and this is where GPU parallelism comes from.

### Warp — A group of threads

GPUs execute threads more efficiently when they are grouped together into batches called warps. Computers love when data comes in a batch and we prefer batching here also, by grouping threads into warps and then using these warps in calculations.

Usually, **1 warp = 32 threads** — this allows the GPU to become faster by allowing us to chunk calculations and process them batch-wise rather than one-by-one.

### Block — A group of warps

A lot of these warps are brought together in a group and that group is called Block. Usually 8 of these warps come together to form a gang and it's named a Block.

Mathematically: **1 Block → 8 Warps → 256 threads**

### Streaming Multiprocessor (SM)

An SM is the actual execution unit inside a GPU. When blocks are assigned to an SM, the SM provides all the hardware required to execute them efficiently. This includes:

- **Schedulers** → decide which threads or warps should run
- **Registers** → ultra-fast temporary storage for calculations
- **Shared Memory** → fast memory shared between threads inside a block
- **Cores** → perform the actual mathematical operations

You can think of an SM as a small factory that takes blocks of work and executes them massively in parallel. This is where the real computation happens inside a GPU. A GPU has got a lot of SMs working together to solve the calculation and get it done by parallelizing it.

## The Execution of Code

The moment you click run, these things happen in the machine. Sticking to our vector addition problem:

1. **CPU reads stuff:** The code is read by the CPU and it understands that you want to run this code on the GPU and passes it to the GPU (this is technically called CPU overhead and a lot of performance engineers hate it). The line in your code file tells the GPU to run on the number of blocks and threads: `matrix_addition<<<blocks, threads>>> (x,y,z)`

2. **GPU creates threads:** The GPU then creates thousands of threads together and all the threads get a specialized task.

3. **Threads are grouped into blocks:** These threads are grouped into blocks, and these blocks become the units scheduled onto the GPU.

4. **Block Assigning:** These blocks are now assigned to Streaming Multiprocessors that are like mini-factories waiting to do all these calculations. The GPU scheduler distributes these blocks to the available SMs.

5. **Actual Calculation Happens:** Inside the SM: data is fetched from memory, cores perform the addition, results are stored back into memory. Since thousands of threads run together, millions of additions can happen simultaneously.

6. **Warp-level latency hiding:** The SM can rapidly switch between multiple warps. So while one warp waits for memory, another warp can continue execution. This helps hide memory latency and keeps the GPU busy.

7. **Output is returned:** Once all threads finish execution, the final output vector z contains all the computed results. The GPU signals completion back to the CPU. And the program continues normally or ends.

## Why GPUs Changed Computing Forever

At first glance, a GPU just looks like a device made for gaming. But internally, it is actually a giant parallel computation machine.

Instead of trying to solve one difficult problem very fast like a CPU, a GPU solves millions of small problems together. When you train an LLM, render a game, or run a massive simulation, you are essentially feeding millions of tiny calculations into thousands of GPU threads running together simultaneously.

And underneath all the complexity, the execution still follows the same hierarchy:

```
Thread → Warp → Block → SM → GPU
```

That is the foundation of modern GPU computing. Once you understand this hierarchy, you stop seeing GPUs as magic black boxes. You start seeing them for what they really are: massive factories built for parallel computation.
