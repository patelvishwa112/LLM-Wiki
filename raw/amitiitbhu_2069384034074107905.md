# How does vLLM work?

**Amit Shekhar** (@amitiitbhu), Founder @ Outcome School — X Article, Jun 2026. Full text via Playwright.

---

## Outline

Serving LLMs → prefill/decode/KV cache → memory bottleneck → naive over-reservation & fragmentation → vLLM (PagedAttention + continuous batching) → OpenAI-compatible API → real-world use (chat, agents).

## Serving

Many users → serving engine → model on GPU. Goal: max users, min latency, limited GPU RAM.

## Prefill / decode / KV cache

Tokens; prefill reads full prompt; decode one token at a time. KV cache stores per-token K/V notes for reuse — grows with sequence length, lives in GPU memory.

## Naive serving problems

- Reserves max-length contiguous KV per request → **over-reservation**
- Short answers leave huge reserved holes → **fragmentation** → can't admit new requests

## PagedAttention

OS-style **pages**: fixed-size blocks (e.g. 16 tokens), **block table** maps logical sequence → physical blocks (non-contiguous). Allocate on demand; free on completion.

**Sharing**: identical prefixes (system prompts); beam search shared prefix blocks.

## Continuous batching

Static batching waits for slowest request → idle slots. Continuous: at each decode step, finished requests swapped out, new ones in → GPU stays busy. Pairs with instant block free from PagedAttention.

## API & benefits

OpenAI-compatible server. Higher throughput, utilization, lower $/request, same model quality.

## Real world

High-traffic chat; **agent systems** (repeated instruction prefix + many short steps).
