# Building the Harness Around Our Coding Agents: Eight Failure Modes, Eight Pillars

Teams building with AI usually end up building two products: the thing they ship, and the system around their agents that makes them useful in building the thing they ship.

We built such a system to help us ship Nimbalyst. We call it our team harness. This post is about what we learned from doing it.

## What a harness is

A harness is the durable layer around a model: instructions, tools, permissions, context, and verification.

Claude Code and Codex are harnesses in this sense. Each wraps a model with a system prompt, a tool surface, a permission model, and an execution loop. Anthropic and OpenAI own that layer.

Your team owns the next layer up: the workspace where agents do product work alongside you, with your files, tasks, diagrams, diffs, and decisions.

Almost nothing in a good harness is novel. It is mostly other people's parts assembled around your project: Claude Code, Codex, MCP, Playwright, a tracker, a diagramming tool, an editor, a test runner, your repository, your docs.

## The Eight Pillars

### 1. Context — Goal: know the project.
**Failure mode:** the agent doesn't know your codebase, rules, decisions, or conventions.
**Solution:** Code, specs, plans, mockups as local files. Architecture diagrams as Excalidraw. Decisions captured as tracker items. Bug histories searchable. CLAUDE.md/AGENTS.md at session start. Path-scoped rules. Skill system for recurring jobs. Persistent memory across sessions.

### 2. Provenance — Goal: trace the why.
**Failure mode:** the agent can't traverse links between artifacts.
**Solution:** Typed link graph between tracker items, plans, specs, diagrams, mockups, sessions, diffs, files, commits, decisions. Decision tracker items. Bug tracker items filed before fix code. MCP surface for graph traversal.

### 3. Capability — Goal: act and observe.
**Failure mode:** agent trapped in text channel, asks user to run every command.
**Solution:** MCP tools for logs, database queries, app control. Screenshot tools. Playwright UI loop. Sandboxed shell for tests/scripts. Extension SDK for custom MCP tools. Agent verifies its own changes.

### 4. Workflow — Goal: reuse the arcs.
**Failure mode:** agent reinvents how to do every task.
**Solution:** Repo-local slash commands (.claude/commands/). Plan-then-execute arc. Investigate/design/implement progression. Subagents for exploration/planning/implementation. Skill system.

### 5. Restraint — Goal: stay in bounds.
**Failure mode:** agent does something dangerous faster than you expected.
**Solution:** Path-scoped rules blocking edits. Hard rules from past incidents. Per-tool permission scopes. Approval flows for destructive/costly actions. Workspace trust modes. Durable audit trail.

### 6. Verification — Goal: prove the fix.
**Failure mode:** agent hallucinates "fixed" without proof.
**Solution:** Failing-test-first discipline. Vitest unit suite. Playwright E2E specs. AI tool simulator for faking tool calls. Fast type checks. Agent must show change works end-to-end.

### 7. Visual Interface — Goal: show the work.
**Failure mode:** decisions and artifacts vanish into chat text.
**Solution:** Workspace where mockup, diff, tracker sit side by side. Markdown editor with red/green diffs. First-class diagram/mockup editors. Inline charts and screenshots. Approval gates with diff view. Voice channel. Team handoff posts.

### 8. Coordination — Goal: track every agent.
**Failure mode:** human loses track of parallel agents.
**Solution:** Kanban board of sessions. Workstreams grouping related sessions. Meta-agent to spawn/supervise others. Git worktrees for isolation. Hand-off briefs. File-edit history tied to sessions.

## Worked Example: One bug, all eight pillars

A sync bug where tracker items showed empty bodies after restart. How the pillars worked together:
- Context loaded CLAUDE.md, sync architecture doc, path-scoped rules
- Provenance: bug linked to 4 prior failed fix sessions
- Capability: queried DB, watched sync logs, found silently swallowed error
- Workflow: /investigate → /design → /implement
- Restraint: read-only prod access, approval-gated commits
- Verification: failing Playwright spec first, turned green only when fixed
- Visual: inline test output, screenshot, interactive merge prompt
- Coordination: 5th session in workstream, inherited all prior context

## Recommendations

**First harness this week:** Do boring parts first — they compound fastest. Specs/plans/diagrams in files. Root instruction file + path-scoped rules. At least 3 capability tools. Approval gates. Linked tickets/docs/files/sessions.

**Already have a harness:** Treat it as a product your team ships to itself. Pick a percentage of AI effort for harness improvement. Multi-agent coordination is a second-quarter problem.

**Own your harness:** Instructions, rules, tool definitions, links, audit logs, skills — if you can't read/edit/version it and point a different agent at it, it's not yours.

**Keep it portable:** When a new coding agent arrives, point it at the same harness without rebuilding.

**Test against failures:** Can you name a recurring failure that doesn't map to a pillar? Framework is missing something. Can you collapse two pillars without bringing a failure back? You have one too many.
