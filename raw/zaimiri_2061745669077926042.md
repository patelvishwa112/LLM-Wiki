## Post

> The useful thing about /goal in Hermes is very simple:
> It turns a chat message into a standing job.
> A normal prompt is disposable. You ask for a thing, the model answers, then it waits for you to steer the next move.
> /goal changes the rhythm.
> You tell Hermes what finished looks like. Hermes starts working, checks whether the job is actually done after each turn, and keeps going until the goal is achieved, paused, cleared, or it hits the turn budget.
> That sounds small until you use it for real work.

## What Hermes is doing under the hood

Hermes stores the goal as session state. After each assistant turn, a separate lightweight judge checks the last output against the goal. If the judge thinks the goal is still incomplete, Hermes automatically continues the same session with the goal still attached.

You do not have to type "continue." You do not have to remember where the agent left off. You do not have to re-explain the task every time the model stops too early.

The standing goal stays there until one of five things happens:
- Hermes achieves it
- You pause it
- You clear it
- You interrupt with a real user message
- It reaches the configured turn budget (default 20 turns)

If you want to keep going, `/goal resume` resets the counter and continues. If the work is wrong, `/goal clear` removes the contract.

"/goal is not a magic autonomy button. It is a control loop. Goal. Work. Judge. Continue. Verify. Stop."

## The commands

- Set a goal: `/goal <description>`
- Check state: `/goal status`
- Pause: `/goal pause`
- Resume: `/goal resume`
- Clear: `/goal clear`
- Add requirement while running: `/subgoal <criteria>`

/subgoal is underrated — sometimes the first goal is directionally right but halfway through you realize the agent needs a tighter boundary. Hermes persists subgoals with the goal, and the judge considers all of them before marking done.

## Where /goal is best

Good use cases:
- fix a failing test suite
- clean up a messy doc folder
- inspect a repo and write a grounded plan
- run a research pass and produce a sourced brief
- migrate repeated prompts into a reusable skill
- debug a tool until the command works
- create a draft, lint it, revise it, and route it for review

Bad use cases:
- one-off questions
- tiny copy edits
- anything where you need to make a taste call every two minutes
- sensitive actions without clear approval boundaries
- tasks where "done" cannot be described

"If you cannot write the done state, do not use /goal yet. Make the task smaller first."

## A real Hermes-style goal

The six-piece pattern:
1. The job
2. The allowed scope
3. The sources to inspect
4. The done criteria
5. The checks to run
6. The final reporting format

Example:
```
/goal
Task: Write a guide to Hermes goal mode.
Sources: Hermes docs for accuracy, this very article for structure only (not wording).
Voice: Zaimiri.
Constraints: No publishing, no scheduling.
Verification: Run slop lint, create Typefully review draft.
Done means: Draft is in Typefully, lint passed, no publication.
Final reply: Short Telegram receipt only.
```

"Most people prompt agents like they are asking a smart intern to 'take a look.' Hermes /goal works better when you treat it like assigning a small job with acceptance criteria."

## Why the judge matters

Agents are confident at the exact moment you should be least relaxed. They will say the test passed when they never ran the test. They will say the file exists when they only planned to create it. They will say the draft is ready when they never pushed it into the draft tool.

/goal does not remove that risk by itself. The judge helps because it asks the boring question after each turn: "Does the last assistant response actually satisfy the goal?"

"The best goals include external checks. Do not make the agent's self-report the proof. Make the proof something Hermes can inspect."

## Telegram makes it more useful

In the CLI, /goal is already good. In Telegram/Signal, it becomes a pocket operator. Send Hermes a goal from your phone, let it work in the background, and check back when the loop finishes or blocks.

"The interface is not the terminal anymore. The interface is a message. The machine still has the files, tools, logs, repo, browser, shell, memory, and skills. You just do not have to sit in front of it for every step."

## The pattern

A strong /goal has six pieces: The job. The allowed scope. The sources to inspect. The done criteria. The checks to run. The final reporting format.

"Do not ask the agent to 'work on the docs.' Give it a finish line."

## What /goal changes

Old workflow: prompt → response → prompt → response → prompt → maybe done

Hermes workflow: goal → work loop → judged continuation → verified result

"It is still early. You still need boundaries. You still need verification. You still need to know when not to automate. But once you get used to defining the finish line once, the normal chat rhythm starts to feel broken. You stop babysitting every turn. You start assigning work."
