# Orchestrating Agents in ADK

**Author:** Franziska Hinkelmann, PhD (@fhinkel)  
**Date:** Jul 13, 2026  
**URL:** https://x.com/fhinkel/status/2076782085516493079  
**Engagement (at capture):** ~1.6K views, 21 likes

---

In my last post I explained agent memory. Memory turns a stateless model into something that can hold a conversation and carry context forward. It is a big step, but it also exposes a new limit. The moment you ask that one agent to book travel, summarize documents, and query a database, you start piling every tool and instruction into a single context window. The agent slowly becomes a generalist that does many things adequately and nothing sharply.

This is a scaling problem, not a model weakness. As the instruction set grows, routing gets fuzzy. The model spends tokens deciding what it is even supposed to be, tool descriptions compete for attention, and unrelated context bleeds into every decision. Anyone who has watched a function balloon until no one can reason about it will recognize the shape of this.

The fix is to stop building one agent that knows everything and start building a team. In ADK, a parent orchestrator coordinates specialized subagents, each with a narrow job and its own small set of tools. The orchestrator's real work is delegation: read the request, pick the right specialist, and hand off.

```python
from google.adk.agents import Agent

flight_agent = Agent(
    name="flight_agent",
    model="gemini-flash-latest",
    instruction="You book and modify flights. Only handle flight requests.",
    tools=[search_flights, book_flight],
)
hotel_agent = Agent(
    name="hotel_agent",
    model="gemini-flash-latest",
    instruction="You book and modify hotels. Only handle lodging requests.",
    tools=[search_hotels, book_hotel],
)
root = Agent(
    name="travel_coordinator",
    sub_agents=[flight_agent, hotel_agent],
    # Auto-injects delegation tools named after each subagent:
    # flight_agent, hotel_agent
)
```

The interesting design decision is how much context these agents share. A single global state pool is convenient, but it reintroduces the very pollution we were trying to escape. Strict isolation keeps each specialist focused, at the cost of deciding exactly what to pass along. A practical rule of thumb: default to isolation, and share only the specific state a subagent needs to do its job.

ADK gives you session state for exactly this handoff. The orchestrator writes the slice a specialist needs, rather than dumping raw tool output back into its own window.

```python
# In the coordinator, before delegating:
ctx.state["trip:destination"] = "Lisbon"
ctx.state["trip:dates"] = "2026-09-01/2026-09-08"
# The hotel_agent reads only these keys, not the full conversation.
```

Clean boundaries keep each agent sharp and your token budget honest.

If you want to build this end to end, the Build Multi-Agent Systems with ADK codelab walks through delegation, session state, and workflow agents like SequentialAgent and ParallelAgent.
