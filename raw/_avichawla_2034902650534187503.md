# KV Caching in LLMs, Clearly Explained

Avi Chawla (@_avichawla) — Mar 20, 2026

**Observation:** First token from ChatGPT/Claude is slow; rest stream fast → **KV caching** for faster inference.

## Part 1: Token generation

Transformer produces hidden states → logits; only **last token logits** used for sampling. Only **last hidden state** needed for next token; others are intermediates.

## Part 2: Attention (last token)

Per layer: Q, K, V. Last row of attention needs:
- Q from latest token
- All K and V in sequence

## Part 3: Redundancy without cache

Token 50 needs K,V for 1..50; token 51 needs 1..51. K,V for 1..49 unchanged but recomputed → **O(n) redundant work per step**, **O(n²)** over generation.

## Part 4: KV cache fix

Per new token:
1. Compute Q,K,V for new token only
2. Append K,V to cache
3. Attention: new Q vs full cached K,V

Projections to K,V once per token; attention still O(seq) but avoids repeat projections. **~5× speedup** in practice (author).

## Part 5: TTFT

**Prefill:** process full prompt, fill cache — most compute-heavy. **Decode:** one token per step with warm cache. **Time-to-first-token** = prefill cost. Longer prompts → longer wait. Related optimizations: chunked prefill, speculative decoding, **prompt caching** (provider-level).

## Part 6: Memory tradeoff

Cache stores per-layer K,V per token. Example: Qwen 2.5 72B (80 layers, 32K ctx, dim 8192) → several GB per request; at scale can exceed weights. **GQA/MQA** share KV heads to cut memory. Longer context doubles KV per request → fewer concurrent users. **Paged Attention** (author cross-links separate article, Mar 11).

## tl;dr

Compute K,V once, reuse; vLLM, TGI, TensorRT-LLM build on this. Cost = GPU memory at scale.
