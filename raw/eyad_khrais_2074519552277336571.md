# How to become an applied AI engineer

**Author:** Eyad (@eyad_khrais), Varick Agents  
**Posted:** Jul 7, 2026

Guide for software engineers transitioning to Applied AI Engineering: deterministic → probabilistic thinking, then evals, harness engineering, and multi-agent coordination.

## Software engineering vs AI engineering

Traditional SWE is deterministic logic; applied AI wraps a non-deterministic LLM API — same input, different outputs. Job shifts from only building software to **measuring** whether the system behaves as intended (**evals**).

## Evals

Give agent a task, let it run, grade result + trajectory.

1. **Outcome grading** — final state vs expected (e.g. invoice placement, duplicate flags).
2. **Trajectory grading** — ordered tool-call log: arguments, fields touched, dangerous paths even when outcome is correct (e.g. bank detail changes while classifying invoice).

Deterministic checks for safety; judge model + rubric for quality. Report **outcome and trajectory scores separately** (95% correct + 4% forbidden-field touches ≠ production-safe).

Resources: Anthropic eval course, Lenny's eval guide, Hamel's FAQ (author suggests Lenny + Hamel before hands-on course).

## Harness engineering

Model alone ≠ agent. Harness = tool execution (JSON request → validate → execute → text result), context management (window budget, summarize/drop), state/memory (outside model in DB/files), guardrails (permissions, validation, human routing), agent loop (context → model → execute → update → repeat).

Context refs: OpenAI context management, practical context guide, Anthropic context engineering; Arize engineer talk on context.

## Multi-agent = distributed systems

Splitting into research/plan/execute/review agents → shared state races, stale reads, duplicate actions.

Patterns:
- **Single-writer** per important state (enforce at tools).
- **Idempotency keys** on mutating tools (Stripe-style).
- **Preconditions on writes** (update only if status still Pending).
- **Explicit hand-offs** — orchestrator + typed messages, not discovery.

## TL;DR

Model supplies intelligence; engineer supplies reliability: measurement (evals), operating environment (harness), coordination (distributed patterns). Extends existing SWE skills.
